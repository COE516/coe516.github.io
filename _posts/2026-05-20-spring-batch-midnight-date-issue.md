## 문제 상황

운영 개발을 하다 보면 평소에는 멀쩡한데 **특정 날만 이상하게 동작하는 배치**를 만날 때가 있습니다.

이번 글은 실제 운영 중 겪었던 날짜 계산 버그 사례입니다.

- 23:30에 시작하는 배치
- 날짜 기준으로 데이터를 조회하는 로직
- 코드 중간중간 `LocalDateTime.now()` 호출
- 처리 시간이 길어질 수 있는 구조

이 조합이라면 한 번쯤 점검해 볼 만합니다.

## 평소에는 문제 없던 배치

당시 배치는 매일 23:30에 실행되는 법인카드 수집 배치였습니다.

흐름은 단순했습니다.

```text
23:30 배치 시작
  ↓
수집 대상 일자 계산
  ↓
카드 승인 데이터 조회
  ↓
GW 적재
```

예를 들어 `2026-03-16 23:30`에 실행됐다면, 배치 전체가 `2026-03-16` 기준으로 동작해야 했습니다.

대부분의 날에는 이 로직이 문제 없이 돌아갔습니다.

## 자정 넘김 이슈는 왜 발생했을까

문제는 배치 처리 시간이 길어진 날에만 드러났습니다.

```text
23:30 시작
23:55 일부 처리 완료
00:03 아직 처리 중
```

이때 일부 로직이 현재 시간을 다시 조회하고 있었습니다.

```java
LocalDate recvDate = LocalDateTime.now().toLocalDate();
```

코드만 보면 자연스럽지만, 자정을 넘기는 순간 결과가 달라집니다.

- `2026-03-16 23:58` -> `2026-03-16`
- `2026-03-17 00:03` -> `2026-03-17`

즉, **같은 배치 실행 안에서 날짜 기준이 바뀌어 버린 것**입니다.

## 실제 운영에서 나타난 증상

실제 운영에서는 같은 배치 안에서 데이터가 섞였습니다.

- 배치 시작 시각: `2026-03-16 23:30`
- 일부 건: `RECV_DATE = 2026-03-16`
- 일부 건: `RECV_DATE = 2026-03-17`

결과적으로 이런 문제가 이어졌습니다.

- 특정 일자 데이터 누락
- 일부 카드 승인내역 미수집
- 운영 재수집 필요

이런 유형은 더 까다롭습니다. 배치 자체는 실패하지 않고, **정상처럼 끝났는데 데이터 기준만 틀린 상태**가 되기 때문입니다.

## 원인은 `now()` 남발이었다

원인은 복잡하지 않았습니다. 배치 전체의 기준 시간이 없었습니다.

예를 들어 아래 구조는 실행 시점에 따라 결과가 달라집니다.

```java
for (Card card : cardList) {
    LocalDate recvDate = LocalDateTime.now().toLocalDate();
    collect(card, recvDate);
}
```

처리 대상이 많거나 외부 API 호출, 느린 DB 조회, retry 로직이 끼어 있으면 루프가 자정을 넘길 수 있습니다.

그 순간부터 같은 Job 안에서도 어떤 건은 어제 날짜, 어떤 건은 오늘 날짜로 처리됩니다.

핵심은 `now()`가 나쁘다는 게 아니라, **같은 배치 실행에서 기준 시간이 하나로 고정되지 않았다는 점**이었습니다.

## 내가 수정한 방식

해결은 단순했습니다. 배치 시작 시점에 시간을 한 번만 구하고, 그 뒤에는 그 값만 사용하게 바꿨습니다.

```java
LocalDateTime batchStartTime = LocalDateTime.now();
LocalDate recvDate = batchStartTime.toLocalDate();
```

변경 전후를 비교하면 더 명확합니다.

```java
// 변경 전
LocalDate recvDate = LocalDateTime.now().toLocalDate();

// 변경 후
LocalDate recvDate = batchStartTime.toLocalDate();
```

조금 더 명시적으로 관리하고 싶다면 컨텍스트로 묶는 방법도 있습니다.

```java
public class BatchContext {

    private final LocalDateTime batchTime;

    public BatchContext(LocalDateTime batchTime) {
        this.batchTime = batchTime;
    }

    public LocalDate getRecvDate() {
        return batchTime.toLocalDate();
    }
}
```

이렇게 바꾸면 배치가 `23:30`에 시작해 `00:30`, `01:00`까지 이어져도 날짜 기준은 끝까지 동일합니다.

## 테스트에서 놓치기 쉬운 이유

이 버그가 무서운 이유는 개발 환경에서 잘 안 보인다는 점입니다.

보통 테스트는 오후나 오전처럼 일반적인 시간대에 진행합니다. 그러면 `now()`를 몇 번 호출해도 날짜가 바뀌지 않으니 문제가 드러나지 않습니다.

재현하려면 최소한 이런 시나리오가 필요합니다.

- `23:59`에 시작
- 처리 도중 `00:01` 진입
- 같은 실행 안에서 날짜 계산이 여러 번 발생

특히 아래 조건이 있으면 위험도가 더 올라갑니다.

- 대량 데이터 처리
- 외부 API 호출
- 느린 DB 응답
- retry 또는 재시도 로직

## 점검할 코드 패턴

배치 코드를 볼 때는 아래 패턴을 먼저 의심해 보는 편이 좋습니다.

```java
LocalDateTime.now()
new Date()
Calendar.getInstance()
```

문제는 이 API 자체가 아니라, **한 실행 안에서 중간중간 다시 읽는 구조**입니다.

배치라면 가능하면 아래 둘 중 하나로 정리하는 편이 안전합니다.

- Job 시작 시각을 한 번 구해서 전달한다.
- 수집 대상 일자를 Job Parameter나 Context로 고정한다.

## 정리

- 자정 근처에 시작하는 배치는 날짜 기준이 흔들리기 쉽습니다.
- `LocalDateTime.now()`를 중간중간 다시 호출하면 같은 배치 안에서도 결과가 달라질 수 있습니다.
- 배치 기준 시간과 실제 처리 시간은 분리해서 보는 편이 안전합니다.
- 운영에서 안 터지길 기대하기보다, **배치 시작 시점을 고정값으로 설계하는 것**이 더 확실합니다.

배치에서 중요한 것은 "지금 몇 시냐"보다 **이번 실행이 어떤 시각에 시작됐느냐**였습니다.
