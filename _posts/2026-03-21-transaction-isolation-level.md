## 문제 상황

격리 수준은 용어를 외워도 실제로 언제 바꿔야 하는지 감이 잘 안 올 때가 많습니다.

실무에서는 `READ COMMITTED`, `REPEATABLE READ`, `SERIALIZABLE` 같은 이름보다 **지금 어떤 이상 현상을 막고 싶은가**를 먼저 보는 편이 훨씬 낫습니다.

## 먼저 기억할 것

격리 수준은 동시에 실행되는 트랜잭션이 서로의 변경을 어디까지 볼 수 있는지 정하는 기준입니다.

보통 아래 세 가지 현상으로 이해합니다.

- dirty read: 커밋되지 않은 값을 읽음
- non-repeatable read: 같은 조회인데 중간에 값이 바뀜
- phantom read: 같은 조건 조회인데 행 수가 달라짐

중요한 것은 이름 자체보다, 내 비즈니스 로직이 이 현상 중 무엇에 민감한가입니다.

## 왜 기본값을 함부로 올리면 안 될까

격리 수준을 높이면 더 안전해 보이지만, 그만큼 잠금 경합과 처리량 저하 가능성도 커집니다.

특히 아래 상황에서는 비용이 바로 체감되기도 합니다.

- 동시에 많이 쓰는 테이블
- 긴 트랜잭션
- 조회와 갱신이 자주 섞이는 업무

그래서 "문제가 생길 것 같아서" 미리 올리기보다, 실제 이상 현상과 경합 비용을 같이 보는 편이 안전합니다.

## 보통 실무에서는 이렇게 본다

### 기본값을 먼저 신뢰한다

대부분의 서비스는 DB 기본 격리 수준으로도 충분히 동작합니다. 애플리케이션 레벨에서 격리 수준을 자주 바꾸는 것은 생각보다 드문 선택입니다.

### 정말 막아야 하는 현상을 먼저 적는다

예를 들어 아래 질문이 더 중요합니다.

- 같은 트랜잭션 안에서 다시 읽을 때 값이 바뀌면 안 되는가
- 범위 조회 결과가 중간에 달라지면 안 되는가
- 재조회나 낙관적 락으로 풀 수 있는 문제인가

격리 수준 문제처럼 보여도, 실제로는 쿼리 순서나 업데이트 방식 문제인 경우도 많습니다.

### 격리 수준 말고 다른 해법도 같이 본다

모든 정합성 문제를 격리 수준만으로 해결하려고 하면 비용이 커질 수 있습니다.

대신 아래 같은 방법이 더 현실적일 때도 많습니다.

- 짧은 트랜잭션 유지
- 명시적 락 사용
- 낙관적 락 버전 컬럼 사용
- 재시도 가능한 구조로 설계

## 이런 경우에 조정을 고민한다

- 같은 트랜잭션 안에서 재조회 일관성이 중요함
- 재고, 정산처럼 특정 이상 현상을 분명히 막아야 함
- 기본값으로는 재현 가능한 정합성 문제가 확인됨

반대로 문제 재현도 없는데 "더 안전해 보여서" 올리는 것은 대개 좋은 출발점이 아닙니다.

## 정리

- 격리 수준은 이름 암기보다 어떤 이상 현상을 막고 싶은지로 이해하는 편이 낫다.
- 기본값을 바꾸기 전에 실제 정합성 문제와 경합 비용을 같이 봐야 한다.
- 많은 경우 격리 수준 조정보다 트랜잭션 길이, 락 전략, 재시도 설계가 더 효과적이다.
- 격리 수준 변경은 예방 차원이 아니라, 확인된 문제를 해결하는 수단으로 쓰는 편이 안전하다.
