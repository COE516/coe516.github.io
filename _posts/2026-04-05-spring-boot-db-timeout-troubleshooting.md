## 문제 상황

간헐적으로 DB 연결 타임아웃이 나면 HikariCP 설정부터 손대고 싶어질 때가 많습니다. 하지만 실제로는 커넥션 풀 고갈, 느린 쿼리, DB 잠금 대기, 네트워크 지연이 섞여 있는 경우가 더 흔합니다.

그래서 이 문제는 timeout 숫자를 바꾸는 것보다 **어디에서 기다리고 있는지 먼저 분리하는 것**이 중요합니다.

## 먼저 세 층으로 나눠서 본다

보통 아래 세 층으로 나누면 원인이 빨리 좁혀집니다.

- 애플리케이션: 풀 고갈, 커넥션 누수, 긴 트랜잭션
- 네트워크: DNS, 방화벽, 일시 지연
- DB: 세션 수 부족, lock wait, slow query, 과부하

## 애플리케이션에서 먼저 볼 것

### 1. 풀에서 연결을 오래 기다리는가

HikariCP timeout은 "DB 접속 자체 실패"가 아니라, 풀에서 연결을 제때 못 받아서 터지는 경우가 많습니다.

### 2. 연결을 오래 쥐고 있는 쿼리가 있는가

실제로는 느린 쿼리나 긴 트랜잭션 때문에 풀이 돌지 않는 경우가 많습니다.

### 3. 누수가 의심되는가

예외 흐름이나 배치 작업에서 커넥션이 길게 잡혀 있으면 트래픽이 조금만 올라가도 timeout이 빨리 드러납니다.

## DB 쪽에서 같이 볼 것

- 현재 커넥션 수가 한계에 가까운지
- 느린 쿼리가 있는지
- lock wait이 길어지는지
- 특정 시간대에만 부하가 몰리는지

즉 애플리케이션 로그만 봐서는 안 되고, DB 상태를 같이 봐야 "왜 기다리는지"가 보입니다.

## 설정 예시는 이 정도만 시작점이다

```yaml
spring:
  datasource:
    hikari:
      connection-timeout: 3000
```

이 값은 증상을 드러내는 기준일 뿐, 근본 해결책은 아닙니다. 숫자를 늘리면 장애가 늦게 보일 수 있고, 너무 줄이면 일시 지연에도 바로 실패할 수 있습니다.

## 흔한 실수

- timeout만 늘려서 임시로 덮음
- 느린 쿼리 확인 없이 풀 설정만 바꿈
- DB lock wait과 애플리케이션 timeout을 따로 봄
- 네트워크 이슈를 애플리케이션 설정 문제로만 봄

## 짧은 점검 순서

1. 풀에서 대기 중인지 확인한다.
2. 오래 잡고 있는 쿼리와 트랜잭션을 본다.
3. DB 세션 수, slow query, lock wait을 같이 본다.
4. 그 다음에 timeout 값을 조정한다.

## 정리

- DB 연결 타임아웃은 설정 한 줄로 끝나는 문제가 아니다.
- 먼저 애플리케이션, 네트워크, DB 중 어디에서 기다리는지 나눠야 한다.
- timeout 값 조정보다 긴 쿼리, 누수, DB 부하 확인이 먼저다.
- 숫자를 바꾸기 전에 원인을 분리하면 불필요한 튜닝을 훨씬 줄일 수 있다.
