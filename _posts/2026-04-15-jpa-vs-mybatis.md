## 문제 상황

새 프로젝트를 시작할 때 JPA와 MyBatis 중 무엇을 고를지 항상 논쟁이 생깁니다.

하지만 실제로는 "무엇이 더 좋으냐"보다 **어떤 종류의 작업이 많은 서비스냐**가 더 중요합니다.

## JPA가 잘 맞는 경우

- 도메인 중심 모델링이 중요함
- 객체 그래프 탐색이 많음
- CRUD 비중이 큼
- 빠르게 기본 저장/조회 구조를 만들고 싶음

즉 엔티티 중심으로 비즈니스 로직을 다루는 흐름과 잘 맞습니다.

## MyBatis가 잘 맞는 경우

- SQL 제어가 매우 중요함
- 복잡한 조회 쿼리가 많음
- 쿼리 튜닝 가시성이 중요함
- 화면/리포트용 조회가 복잡함

즉 조회 성능과 SQL 명시성이 핵심일 때 더 직관적인 경우가 많습니다.

## 실무에서 많이 갈리는 지점

### CRUD 중심인가, 조회 중심인가

주문, 회원, 게시글처럼 전형적인 CRUD 흐름이 많으면 JPA가 생산성에서 유리할 수 있습니다.

반대로 통계, 집계, 복잡한 조인 조회가 많으면 MyBatis 쪽이 다루기 편한 경우가 많습니다.

### 팀이 무엇에 익숙한가

기술 자체보다 팀이 익숙한 방식도 중요합니다.

JPA를 잘 모르는 팀이 억지로 복잡한 연관관계를 다루면 생산성이 떨어질 수 있고, 반대로 SQL 관리가 약한 팀이 MyBatis를 쓰면 쿼리 관리 부담이 커질 수 있습니다.

## 한쪽만 고집할 필요는 없다

실무에서는 한 프로젝트 안에서도 역할을 나눠 쓰는 경우가 있습니다.

- 기본 도메인 저장/수정은 JPA
- 복잡한 조회는 MyBatis

처럼 가져가면 각 도구 장점을 살리기 쉬울 때가 있습니다.

## 정리

- CRUD 중심이면 JPA가 생산성과 구조화에 유리할 수 있다.
- SQL 제어와 복잡한 조회가 핵심이면 MyBatis가 더 직관적일 수 있다.
- 팀 경험과 서비스 성격을 같이 봐야 한다.
- 중요한 건 유행보다, 서비스에서 자주 다루는 작업 종류에 맞게 고르는 것이다.
