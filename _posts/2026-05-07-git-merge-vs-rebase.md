## 문제 상황

Git을 쓰다 보면 `merge`와 `rebase`를 둘 다 보게 됩니다.

둘 다 브랜치를 합치는 데 쓰인다는 말은 들었는데, 막상 실무에서 보면 "그래서 둘 중 뭘 써야 하지?", "히스토리가 어떻게 달라지는 거지?"에서 헷갈리는 경우가 많습니다.

특히 팀 프로젝트에서는 이 차이를 대충 이해하고 쓰면 커밋 히스토리가 복잡해지거나, 이미 공유한 브랜치를 다시 밀어 넣는 상황이 생길 수 있습니다.

## 먼저 한 문장으로 정리하면

아주 단순하게 보면 이렇게 이해하면 됩니다.

- `merge`: 브랜치를 **있는 그대로 합친다**
- `rebase`: 내 브랜치 커밋을 **다른 브랜치 위에 다시 쌓는다**

둘 다 최종 코드 결과는 비슷할 수 있지만, **커밋 히스토리가 만들어지는 방식**이 다릅니다.

## merge는 어떻게 동작할까

예를 들어 `main` 브랜치에서 `feature` 브랜치를 따서 작업했다고 가정해 보겠습니다.

작업 중간에 `main`에도 새 커밋이 추가되면 히스토리는 대략 이렇게 됩니다.

```text
A---B---C main
     \
      D---E feature
```

이 상태에서 `feature`를 `main`에 merge하면 보통 이런 모양이 됩니다.

```text
A---B---C------M main
     \        /
      D------E feature
```

여기서 `M`은 merge commit입니다.

즉, 두 브랜치가 갈라져서 각각 어떻게 흘러왔는지를 **기록으로 남긴 채 합치는 방식**이라고 보면 됩니다.

## rebase는 어떻게 동작할까

같은 상황에서 `feature` 브랜치를 `main` 위로 rebase하면 모양이 달라집니다.

```text
A---B---C main
         \
          D'---E' feature
```

원래 `feature`의 `D`, `E` 커밋이 그대로 붙는 것이 아니라, `main`의 최신 커밋 `C` 뒤에 **새로 다시 만들어진 커밋** `D'`, `E'`로 쌓입니다.

즉 rebase는 "브랜치를 합친다"기보다, **내 작업을 최신 기준선 위로 다시 정렬한다**는 느낌으로 이해하면 쉽습니다.

## 왜 헷갈릴까

실무에서 헷갈리는 이유는 둘 다 결국 브랜치 차이를 반영해 최종 코드를 합치기 때문입니다.

그래서 파일 결과만 보면 비슷해 보일 수 있습니다.
하지만 아래는 분명히 다릅니다.

- 히스토리가 갈라진 흔적을 남기는가
- merge commit이 생기는가
- 기존 커밋 SHA가 유지되는가
- 공유 브랜치에서 안전한가

특히 마지막 항목이 중요합니다.

## merge의 장점

### 1. 히스토리를 있는 그대로 보존한다

브랜치가 언제 갈라졌고 어떻게 합쳐졌는지 흐름이 남습니다.

여러 사람이 동시에 작업하는 팀에서는 "이 기능이 어떤 브랜치에서 합쳐졌는지"를 추적하기 더 쉬운 경우가 있습니다.

### 2. 이미 공유된 브랜치에도 비교적 안전하다

merge는 기존 커밋을 바꾸지 않고 새 merge commit을 추가하는 방식이므로, 원격에 이미 올라간 브랜치에서도 부담이 적습니다.

즉 팀원이 같이 보고 있는 브랜치라면 merge가 더 안전한 선택인 경우가 많습니다.

### 3. 기록이 명시적이다

큰 기능 단위로 합쳐질 때 merge commit이 남아 있으면 "여기서 한 덩어리 작업이 들어왔구나"를 한눈에 보기 좋을 때도 있습니다.

## rebase의 장점

### 1. 히스토리가 더 직선적으로 보인다

merge commit이 계속 쌓이면 로그가 복잡해질 수 있습니다.
rebase를 쓰면 중간중간 갈라진 흔적 없이 비교적 깔끔한 선형 히스토리를 만들 수 있습니다.

### 2. 내 브랜치를 최신 기준에 맞춰 정리하기 좋다

작업 브랜치에서 오래 작업했다면, rebase로 `main` 최신 변경사항을 먼저 반영해 충돌을 미리 정리할 수 있습니다.

이렇게 하면 최종적으로 PR을 합칠 때 흐름이 더 깨끗해질 수 있습니다.

### 3. 불필요한 merge commit을 줄일 수 있다

작은 브랜치를 자주 동기화해야 하는 경우 merge를 반복하면 로그가 금방 복잡해집니다.
이럴 때는 rebase가 보기 좋은 히스토리를 만드는 데 유리합니다.

## 가장 중요한 차이: 커밋을 바꾸는가

이 부분을 기억하면 실수를 많이 줄일 수 있습니다.

- `merge`는 기존 커밋을 그대로 두고 새 커밋을 추가한다.
- `rebase`는 기존 커밋을 새로운 기반 위에서 다시 만든다.

그래서 rebase 후에는 커밋 SHA가 바뀝니다.

이 말은 곧, **이미 다른 사람과 공유한 브랜치에서 rebase를 조심해야 한다**는 뜻입니다.
공유 브랜치를 rebase한 뒤 강제로 push하면, 다른 팀원의 기준선이 흔들릴 수 있습니다.

## 언제 merge를 쓰는 편이 좋을까

아래 같은 경우라면 merge가 더 무난합니다.

- 협업 중인 공용 브랜치인 경우
- 히스토리의 분기와 합류 자체가 의미 있는 경우
- 이미 원격에 공유된 브랜치를 안전하게 합치고 싶은 경우
- 팀 규칙상 merge commit을 유지하는 경우

특히 Git이 익숙하지 않은 팀에서는 merge가 더 예측 가능하고 사고가 적은 편입니다.

## 언제 rebase를 쓰는 편이 좋을까

아래 같은 경우라면 rebase가 잘 맞습니다.

- 아직 나만 쓰는 개인 작업 브랜치인 경우
- PR을 올리기 전에 커밋 흐름을 정리하고 싶은 경우
- `main`의 최신 변경사항을 반영하면서 충돌을 미리 해결하고 싶은 경우
- 선형 히스토리를 선호하는 팀 규칙이 있는 경우

즉 rebase는 **공유 전 정리 작업**에 특히 잘 어울립니다.

## 실무에서 많이 쓰는 안전한 패턴

많은 팀에서는 아래처럼 씁니다.

### 개인 작업 브랜치에서는 rebase

```bash
git checkout feature/login
git fetch origin
git rebase origin/main
```

이렇게 하면 `main` 최신 변경사항 위로 내 작업을 다시 올려서 충돌을 미리 정리할 수 있습니다.

### 최종 반영은 팀 규칙에 따라 merge 또는 squash merge

GitHub PR에서는 merge, squash merge, rebase merge 같은 옵션을 제공하기도 합니다.

중요한 것은 "무조건 rebase가 좋다"가 아니라, **팀이 어떤 히스토리를 원하는지에 맞춰 일관되게 쓰는 것**입니다.

## 충돌은 어느 쪽에서도 날 수 있다

가끔 rebase를 쓰면 충돌이 없어지는 것으로 오해하기도 합니다.

하지만 merge와 rebase는 둘 다 같은 파일을 같은 부분에서 다르게 수정했다면 충돌이 날 수 있습니다.
차이는 충돌이 나는 "시점"과 히스토리 정리 방식에 더 가깝습니다.

- merge: 합치는 순간 한 번에 충돌을 볼 수 있다.
- rebase: 커밋을 하나씩 다시 적용하면서 충돌을 볼 수 있다.

그래서 rebase 충돌은 더 세밀하게 보일 수 있고, 대신 익숙하지 않으면 부담스럽게 느껴질 수도 있습니다.

## 쉽게 비유하면

비유로 보면 조금 더 이해가 쉽습니다.

- `merge`는 두 갈래로 작성한 작업 기록을 **그대로 묶어서 보관하는 방식**에 가깝습니다.
- `rebase`는 내 작업 노트를 최신 본문 흐름에 맞게 **다시 정리해서 이어 붙이는 방식**에 가깝습니다.

하나는 기록 보존에 강하고, 다른 하나는 흐름 정리에 강하다고 보면 됩니다.

## 정리

- `merge`는 브랜치를 있는 그대로 합치며 merge commit이 생길 수 있다.
- `rebase`는 내 커밋을 다른 브랜치 최신 위치 위에 다시 쌓는다.
- merge는 공유 브랜치에서 더 안전하고, rebase는 개인 브랜치 정리에 유리하다.
- 둘 중 무엇이 절대적으로 더 좋다기보다, 팀 협업 방식과 히스토리 관리 기준에 맞춰 선택하는 것이 중요하다.

처음에는 "merge는 보존, rebase는 정리" 정도로 기억해도 충분합니다.
그 기준만 잡혀 있어도 Git 히스토리를 훨씬 덜 헷갈리게 볼 수 있습니다.
