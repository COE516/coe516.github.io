## 문제 상황

초기에는 Entity를 그대로 API 응답으로 내보내고 싶어집니다. 빠르고 코드도 적어 보이기 때문입니다.

## 왜 분리할까

가장 큰 이유는 경계 보호입니다.

- API 형식과 DB 모델을 분리할 수 있음
- 불필요한 필드 노출을 막을 수 있음
- lazy loading 문제를 줄일 수 있음
- 화면 요구사항 변경이 엔티티 변경으로 바로 이어지지 않음

## 언제 특히 필요할까

- 외부 공개 API가 있을 때
- 응답 구조가 자주 바뀔 때
- 연관관계가 복잡할 때
- 보안상 숨겨야 할 필드가 있을 때

## 정리

작은 프로젝트에서는 바로 체감이 안 될 수 있지만, 규모가 커질수록 DTO 분리의 이점이 커집니다.

- Entity는 저장 모델
- DTO는 입출력 모델
- 둘을 분리하면 변경 영향 범위를 줄이기 쉽다.
