## 문제 상황

초기에는 Entity를 그대로 요청이나 응답에 써도 빨라 보입니다. 클래스도 덜 만들고 바로 JSON으로 내보낼 수 있어서 편해 보이기 때문입니다.

하지만 API 요구사항이 조금만 바뀌면 그때부터 DB 모델과 외부 노출 형식이 같이 엮이기 시작합니다.

## 왜 분리할까

핵심은 역할 분리보다 **경계 보호**에 가깝습니다.

- API 형식과 DB 모델을 분리할 수 있음
- 숨겨야 할 필드를 실수로 노출할 가능성을 줄임
- 연관관계 직렬화나 lazy loading 문제를 줄이기 쉬움
- 화면 요구사항 변경이 엔티티 변경으로 바로 이어지지 않음

## 실무에서 바로 드러나는 문제

예를 들어 `Member` 엔티티에 비밀번호, 권한, 주문 목록 같은 값이 같이 붙어 있으면 API 응답으로 그대로 내보내기 시작하는 순간 고민이 생깁니다.

- 이 필드를 정말 외부에 노출해도 되는가
- 연관 객체까지 따라 나가지는 않는가
- 화면마다 필요한 필드가 다른데 엔티티 하나로 버틸 수 있는가

이때 DTO가 없으면 응답 요구사항이 엔티티를 직접 흔들기 시작합니다.

## 언제 특히 분리가 중요할까

### 외부 공개 API가 있을 때

응답 스펙이 계약이 되기 때문에 내부 모델과 분리하는 편이 안전합니다.

### 연관관계가 복잡할 때

엔티티 직렬화는 생각보다 쉽게 lazy loading, 순환 참조, 과한 데이터 노출 문제로 이어질 수 있습니다.

### 화면별 요구사항이 다를 때

목록 화면, 상세 화면, 관리자 화면이 각각 다른 필드를 원하면 DTO가 있는 편이 훨씬 다루기 쉽습니다.

### 요청 검증이 중요할 때

입력 전용 DTO에 검증 규칙을 두면 엔티티에 웹 요청 책임을 섞지 않을 수 있습니다.

## 언제는 단순하게 갈 수 있을까

아주 작은 내부용 프로젝트에서 구조가 단순하고 노출 범위도 좁다면 초반에 체감이 적을 수는 있습니다. 다만 그 경우에도 "지금은 단순해서 괜찮다"와 "항상 엔티티를 바로 써도 된다"는 다릅니다.

## 짧은 기준

- Entity는 저장 모델로 본다.
- DTO는 요청/응답 모델로 본다.
- 외부 노출, 검증, 화면별 응답 차이가 생기면 분리 이점이 커진다.
- 특히 JPA 연관관계가 복잡해질수록 DTO 분리가 훨씬 안전하다.

## 정리

- DTO 분리는 형식적인 계층 놀이가 아니라 경계 보호를 위한 선택이다.
- 엔티티를 바로 노출하면 API 변경과 DB 모델 변경이 쉽게 엮인다.
- 작은 프로젝트에서는 체감이 약할 수 있어도 규모가 커질수록 이점이 커진다.
- 실무에서는 저장 모델과 입출력 모델을 분리해두는 편이 훨씬 덜 흔들린다.
