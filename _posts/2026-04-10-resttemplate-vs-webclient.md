## 문제 상황

외부 API를 붙일 때마다 "이제 RestTemplate은 버려야 하나" 같은 고민이 자주 나옵니다. 하지만 실무에서는 최신 도구를 고르는 문제보다 **지금 서비스가 어떤 방식으로 호출을 처리하는지**가 더 중요합니다.

## 가장 큰 차이

핵심은 실행 모델입니다.

- `RestTemplate`: 동기, 블로킹
- `WebClient`: 비동기/논블로킹 사용 가능

즉 WebClient가 더 현대적일 수는 있어도, 모든 서비스에서 자동으로 더 좋은 선택이 되는 것은 아닙니다.

## RestTemplate이 아직 잘 맞는 경우

아래 조건이면 RestTemplate도 충분히 실용적입니다.

- 일반적인 Spring MVC 서비스
- 외부 호출 수가 아주 많지 않음
- 요청 하나당 외부 API 한두 번 호출하는 수준
- 팀이 reactive 디버깅에 익숙하지 않음

이 경우에는 단순성과 운영 편의성이 더 중요할 수 있습니다.

## WebClient가 값이 커지는 경우

아래 상황이면 WebClient를 더 적극적으로 볼 만합니다.

- 동시 외부 호출이 많음
- 타임아웃, 커넥션, 재시도 제어가 중요함
- 여러 API를 조합하면서 대기 시간을 줄이고 싶음
- 이미 WebFlux나 reactive 흐름을 쓰고 있음

특히 외부 API 여러 개를 동시에 호출해야 하는 구간에서는 선택지가 더 넓어집니다.

## 자주 놓치는 부분

### MVC에서 WebClient를 써도 `block()`으로 끝나는 경우

이 패턴 자체가 무조건 나쁜 것은 아닙니다. 다만 이렇게 쓰면 논블로킹의 이점을 크게 가져가지는 못할 수 있습니다.

### WebClient를 쓰면 무조건 빨라진다고 생각하는 경우

상대 API가 느리면 기다리는 시간 자체는 여전히 존재합니다. 병목이 호출 대상에 있으면 클라이언트 라이브러리만 바꿔도 체감 개선이 작을 수 있습니다.

### 팀이 운영할 수 있는지 안 보는 경우

reactive 체인에서 예외 흐름, 타임아웃, 로깅을 익숙하게 다루지 못하면 오히려 장애 분석이 더 어려워질 수 있습니다.

## 짧은 선택 기준

- MVC + 단순 호출이면 RestTemplate도 충분하다.
- 동시 호출, 커넥션 효율, reactive 흐름이 중요하면 WebClient가 유리하다.
- WebClient를 써도 blocking으로만 끝내면 이점은 제한적일 수 있다.
- 성능 기대보다 팀 운영 가능성을 같이 봐야 한다.

## 정리

- 선택 기준은 최신 여부보다 서비스 호출 패턴이다.
- RestTemplate은 단순한 동기 호출에 아직 실용적이다.
- WebClient는 고동시성, 조합 호출, reactive 환경에서 강점이 크다.
- 결국 잘 맞는 도구는 팀이 이해하고 운영할 수 있는 쪽이다.
