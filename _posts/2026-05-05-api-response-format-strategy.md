## 문제 상황

어떤 API는 바로 데이터만 주고, 어떤 API는 `data`, `message`, `result`를 감싸서 주면 클라이언트가 처리하기 점점 불편해집니다.

특히 성공 응답보다 에러 응답 구조가 제각각이면 프런트엔드나 다른 소비자가 매번 예외 케이스를 따로 처리해야 합니다.

## 중요한 기준

응답 포맷은 복잡함보다 **예측 가능성**이 중요합니다.

보통은 아래 둘 중 하나를 정해서 유지합니다.

- 성공 시 데이터 직접 반환
- 성공/실패 모두 공통 wrapper 사용

핵심은 무엇을 고르든, 같은 종류의 API에서 계속 같은 규칙을 유지하는 것입니다.

## 무엇을 먼저 맞춰야 할까

### 1. 성공 응답 방식

예를 들어 목록 조회, 단건 조회, 생성 응답에서 어떤 구조를 쓸지 먼저 맞춥니다.

### 2. 에러 응답 구조

실무에서는 이쪽이 더 중요할 때가 많습니다.

- 에러 코드
- 사용자 메시지
- 상세 필드 오류

처럼 어떤 필드를 항상 줄지 먼저 정하는 편이 낫습니다.

### 3. 페이지 응답 같은 반복 구조

페이지 응답, 목록 응답, 공통 메타데이터도 같은 규칙을 유지해야 클라이언트가 예측 가능하게 쓸 수 있습니다.

## 예를 들면 이런 식이다

성공 시 데이터를 바로 줄 수도 있고,

```json
{
  "id": 1,
  "name": "item"
}
```

공통 wrapper를 둘 수도 있습니다.

```json
{
  "data": {
    "id": 1,
    "name": "item"
  },
  "message": null
}
```

중요한 것은 어떤 스타일이 더 예뻐 보이느냐보다, 프로젝트 전체에서 계속 같은 기준을 유지하느냐입니다.

## 실무 기준

- 상태 코드는 HTTP 의미를 따른다.
- 에러 본문은 코드, 메시지, 상세 필드를 일정하게 둔다.
- 같은 종류의 API는 같은 응답 규칙을 유지한다.
- 팀 안에서 한 번 정한 규칙은 예외 없이 지키는 편이 낫다.

## 언제 더 주의해야 할까

- 여러 클라이언트가 같은 API를 쓰는 경우
- 에러 처리 공통화가 중요한 경우
- 페이지 응답, 배치 응답처럼 반복 구조가 많은 경우

이럴수록 응답 포맷의 작은 흔들림이 나중에 더 큰 비용이 됩니다.

## 정리

- 응답 포맷은 예측 가능해야 한다.
- 성공 응답보다 실패 응답 통일이 더 중요할 때가 많다.
- wrapper 사용 여부보다, 팀 규칙을 정하고 끝까지 유지하는 편이 더 중요하다.
- 설계가 애매하면 "클라이언트가 이 응답을 일관되게 처리할 수 있는가"를 기준으로 다시 보면 도움이 된다.
