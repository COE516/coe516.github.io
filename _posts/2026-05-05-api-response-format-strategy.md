## 문제 상황

어떤 API는 바로 데이터만 주고, 어떤 API는 `data`, `message`, `result`를 감싸서 주면 클라이언트가 처리하기 점점 불편해집니다.

## 중요한 기준

응답 포맷은 복잡함보다 일관성이 중요합니다.

보통 아래 둘 중 하나를 정해서 유지합니다.

- 성공 시 데이터 직접 반환
- 성공/실패 모두 공통 wrapper 사용

## 무엇이 더 좋을까

정답은 없지만 한 번 정했다면 끝까지 유지하는 편이 중요합니다. 특히 에러 응답 구조는 더더욱 통일돼 있어야 합니다.

## 추천 포인트

- 상태 코드는 HTTP 의미를 따른다.
- 에러 본문은 코드, 메시지, 상세 필드를 일정하게 둔다.
- 페이지 응답이나 목록 응답도 같은 규칙을 유지한다.

## 정리

- 응답 포맷은 예측 가능해야 한다.
- 성공보다 실패 응답 통일이 더 중요할 때가 많다.
- 팀 규칙을 정하고 끝까지 유지하는 편이 낫다.
