## 문제 상황

헬스체크를 단순히 `200 OK` 정도로만 두면 프로세스만 살아 있어도 인프라는 인스턴스를 정상으로 판단합니다.

하지만 실제 운영에서는 앱이 떠 있어도 아래 상황이면 요청을 받게 하면 안 됩니다.

- DB 연결 풀이 이미 고갈된 경우
- 필수 외부 의존성이 아직 준비되지 않은 경우
- 애플리케이션은 떴지만 초기화가 끝나지 않은 경우

그래서 헬스체크는 "살아 있나"와 "트래픽을 받아도 되나"를 나눠서 봐야 합니다.

## liveness와 readiness는 역할이 다르다

### liveness

프로세스가 살아 있는지를 봅니다. 이 값이 깨지면 보통 컨테이너나 프로세스를 다시 띄우는 쪽으로 이어집니다.

### readiness

지금 이 인스턴스에 트래픽을 보내도 되는지를 봅니다. 이 값이 깨지면 재시작보다 **로드밸런서 대상에서 잠시 제외하는 판단**에 더 가깝습니다.

운영에서는 readiness가 더 실무적인 경우가 많습니다. 앱이 죽은 것은 아니지만, 지금 요청을 받으면 실패만 늘어나는 상태를 잡아내기 때문입니다.

## readiness에 무엇을 넣을까

보통 아래 기준으로 생각하면 정리가 쉽습니다.

- 요청 처리에 반드시 필요한 의존성인가
- 잠깐 실패해도 트래픽을 계속 받아야 하는가
- 이 값이 실패했을 때 인프라가 어떤 행동을 하게 되는가

예를 들어 메인 DB는 readiness에 포함할 가능성이 높지만, 참고용 통계 API까지 모두 readiness에 묶어버리면 외부 시스템 일시 장애 때 전체 트래픽이 빠질 수 있습니다.

핵심은 "연결 가능 여부"보다 **이 의존성이 없으면 지금 요청을 정상 처리할 수 있는가**입니다.

## Spring Boot에서는 보통 이렇게 시작한다

Actuator를 쓰면 기본 구성을 빠르게 가져갈 수 있습니다.

```yaml
management:
  endpoint:
    health:
      probes:
        enabled: true
  endpoints:
    web:
      exposure:
        include: health
```

이렇게 두면 보통 아래 엔드포인트를 기준으로 인프라와 연결할 수 있습니다.

- `/actuator/health/liveness`
- `/actuator/health/readiness`

쿠버네티스를 쓰든, 별도 로드밸런서를 쓰든 중요한 것은 엔드포인트 주소보다 **실패 기준을 어떻게 설계했는가**입니다.

## 흔한 실수

- liveness와 readiness를 같은 의미로 사용함
- readiness에 너무 많은 외부 의존성을 넣음
- 반대로 핵심 DB나 메시지 브로커를 readiness에서 빼버림
- 장애 시 재시작이 필요한지, 트래픽 차단이 필요한지 구분하지 않음

헬스체크는 모니터링 정보이기도 하지만, 실제로는 인프라의 동작을 바꾸는 운영 신호라는 점을 잊지 않는 편이 좋습니다.

## 정리

- 헬스체크는 단순 ping이 아니라 트래픽 제어 신호다.
- liveness는 생존 여부, readiness는 요청 수신 가능 여부에 가깝다.
- readiness에는 "지금 이 요청을 정상 처리할 수 있는가" 기준으로 의존성을 넣는다.
- 설정 자체보다 실패 시 인프라가 어떤 행동을 하게 되는지까지 같이 설계해야 한다.
