## 문제 상황

`@Component`, `@Service`, `@Repository`는 다 빈으로 등록되기 때문에 아무 데나 붙여도 되는 것처럼 보일 때가 있습니다. 실제로 동작만 놓고 보면 큰 차이가 없어 보여 더 헷갈립니다.

하지만 실무에서는 "등록되느냐"보다 **이 클래스가 무슨 책임을 가지는지 바로 읽히느냐**가 더 중요합니다.

## 공통점

셋 다 component scan 대상입니다. 즉 기본적으로는 스프링 컨테이너에 빈으로 등록됩니다.

그래서 차이는 등록 기능 자체보다, 역할을 드러내는 의미와 일부 부가 동작에 있습니다.

## 어떻게 나누면 될까

### `@Component`

계층 의미를 강하게 주고 싶지 않은 일반 컴포넌트에 씁니다.

- 외부 API 클라이언트
- 보조 변환기
- 공통 유틸 성격의 빈
- 특정 계층에 딱 묶이지 않는 어댑터

### `@Service`

비즈니스 규칙과 흐름을 담는 서비스 계층에 붙입니다.

예를 들면 주문 생성, 결제 처리, 회원 상태 변경처럼 "무엇을 한다"가 중요한 곳입니다.

### `@Repository`

DB 접근 책임이 있는 계층에 붙입니다.

단순 의미 표시만이 아니라, 스프링의 예외 변환과도 연결되기 때문에 영속성 접근 코드에는 이쪽이 더 자연스럽습니다.

## 자주 꼬이는 부분

### 유틸 클래스에도 무조건 `@Service`를 붙이는 경우

서비스 계층이 아닌데도 관성적으로 `@Service`를 붙이면 코드 읽는 사람이 책임을 오해하기 쉽습니다.

### DB 접근이 있는데 `@Component`로만 두는 경우

동작은 할 수 있어도 의도가 흐려지고, 예외 변환 같은 스프링 의미도 놓치기 쉽습니다.

### 애노테이션만 맞고 실제 책임은 섞여 있는 경우

이게 더 흔한 문제입니다. 이름표보다 중요한 것은 진짜로 비즈니스 로직, DB 접근, 보조 기능이 분리되어 있는가입니다.

## 짧은 기준

- 공통 보조 컴포넌트면 `@Component`
- 비즈니스 흐름이면 `@Service`
- 영속성/DB 접근이면 `@Repository`
- 핵심은 빈 등록보다 책임을 읽기 쉽게 나누는 것

## 정리

- 셋 다 빈으로 등록되지만 의미는 다르다.
- 차이는 기술 기능보다 역할 표현에서 더 크게 드러난다.
- 특히 `@Repository`는 DB 접근 책임을 분명히 하는 데 유용하다.
- 실무에서는 애노테이션 이름보다 클래스 책임이 섞이지 않게 유지하는 편이 더 중요하다.
