## 문제 상황

`application.yml`에 값을 넣어뒀는데 실행하면 다른 값이 적용되는 경우가 있습니다. 로컬에서는 잘 되는데 운영에서만 이상하거나, 파일을 고쳤는데 반영이 안 되는 느낌이 들 때도 많습니다.

이 문제는 대부분 파일 내용보다 **어디에서 더 높은 우선순위로 덮어썼는지**를 놓쳐서 생깁니다.

## 기본 이해

Spring Boot는 한 파일만 읽지 않습니다. 여러 소스에서 값을 읽고, 우선순위가 높은 값으로 최종 결정합니다.

대표적으로 영향을 주는 것은 아래입니다.

- 기본 `application.yml`
- profile별 설정 파일
- 환경변수
- JVM 옵션
- 커맨드라인 인자

즉 "파일에 적어뒀다"는 사실만으로는 실제 적용값을 설명할 수 없습니다.

## 실무에서 자주 헷갈리는 경우

### 1. profile 파일이 덮어쓴 경우

`application.yml`의 값을 `application-prod.yml`이 다시 덮을 수 있습니다.

### 2. 환경변수가 더 위에 있는 경우

컨테이너 환경에서는 yml보다 환경변수가 우선되는 경우가 많습니다. 특히 쿠버네티스나 배포 플랫폼에서는 이쪽을 먼저 의심하는 편이 빠릅니다.

### 3. 실행 옵션으로 덮어쓴 경우

배포 스크립트나 CI 설정에서 개별 프로퍼티를 넘기고 있으면, 파일을 바꿔도 결과가 안 바뀐 것처럼 보일 수 있습니다.

## 문제를 볼 때 순서

### 먼저 활성 profile을 본다

설정 이상은 파일 내용보다 "지금 어떤 profile로 떴는가"를 먼저 보는 편이 빠릅니다.

### 그다음 환경변수를 본다

운영에서는 생각보다 많은 값이 환경변수로 주입됩니다.

### 마지막으로 실행 옵션을 본다

커맨드라인 인자나 JVM 옵션은 우선순위가 높아서 실제 원인을 가리는 경우가 많습니다.

## 추적 팁

- 현재 활성 profile을 먼저 확인한다.
- 환경변수 주입 여부를 확인한다.
- 실행 커맨드에 옵션이 있는지 본다.
- actuator나 로그로 최종 적용값을 추적한다.

핵심은 설정 파일을 하나 더 읽는 것이 아니라, 최종 값을 어디서 만들었는지 따라가는 것입니다.

## 정리

- 설정 문제는 파일 하나만 봐서는 잘 안 풀린다.
- `application.yml`보다 더 높은 우선순위 소스를 항상 의심해야 한다.
- 운영 환경일수록 환경변수와 실행 옵션 영향이 크다.
- 문제가 생기면 파일 수정부터 하기보다 활성 profile과 최종 적용값부터 확인하는 편이 빠르다.
