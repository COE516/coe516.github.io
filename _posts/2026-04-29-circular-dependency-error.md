## 문제 상황

서비스를 나누다 보면 어느 순간 A가 B를 주입받고, B도 다시 A를 주입받는 구조가 생깁니다. 예전에는 우연히 넘어가던 코드가 Spring Boot 버전이나 설정이 바뀌면서 바로 실패하기도 합니다.

이때 중요한 것은 "어떻게 우회할까"보다 **왜 두 클래스가 서로를 꼭 알아야 하는 구조가 됐는가**입니다.

## 왜 문제일까

순환 참조는 단순 주입 기술 문제가 아니라 책임 경계가 흐려졌다는 신호인 경우가 많습니다.

```java
@Service
class OrderService {
    private final PaymentService paymentService;
}

@Service
class PaymentService {
    private final OrderService orderService;
}
```

이 구조는 보통 한쪽이 가져가야 할 흐름이 양쪽에 섞였다는 뜻입니다.

## 흔한 원인

- 두 서비스가 같은 비즈니스 흐름을 양쪽에서 다 처리함
- 조회와 상태 변경 책임이 섞임
- 공통 로직을 분리하지 않고 서로 호출함
- 후처리 성격의 작업까지 즉시 직접 호출로 연결함

## 보통 이렇게 끊는다

### 1. 공통 책임을 별도 서비스로 분리

가장 자주 쓰는 방법입니다. 양쪽에서 필요하던 공통 로직을 제3의 서비스로 빼면 의존성 방향이 정리됩니다.

### 2. 호출 방향을 한쪽으로 고정

주문이 결제를 호출할지, 결제가 주문을 호출할지 한쪽만 남기고 역방향을 없앱니다.

### 3. 이벤트로 분리

즉시 강결합이 필요 없는 후처리라면 이벤트로 끊는 편이 자연스러울 수 있습니다.

## 임시 우회는 조심

`@Lazy`를 쓰면 당장 부팅은 통과할 수 있습니다. 하지만 대부분은 구조 문제가 남아 있어서 나중에 다시 읽기 어려운 코드가 됩니다.

급한 대응이 아니라면 우회보다 책임 재정리가 먼저입니다.

## 짧은 기준

- 서로가 서로를 꼭 알아야 하는지 먼저 의심한다.
- 공통 책임이 있으면 별도 서비스로 분리한다.
- 호출 방향은 한쪽으로 고정한다.
- `@Lazy`는 설계 해결책보다 임시 우회에 가깝게 본다.

## 정리

- 순환 참조는 주입 기술 문제보다 설계 문제인 경우가 많다.
- 해결의 핵심은 의존성을 숨기는 것이 아니라 방향을 다시 정리하는 것이다.
- 공통 책임 분리와 이벤트 분리가 가장 자주 쓰이는 해법이다.
- 우회보다 구조를 고치는 편이 장기적으로 훨씬 낫다.
