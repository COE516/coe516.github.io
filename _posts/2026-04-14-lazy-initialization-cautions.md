## 문제 상황

애플리케이션 시작 시간이 느릴 때 Lazy Initialization을 켜면 빨라지는 것처럼 보일 수 있습니다.

하지만 이 설정은 단순한 성능 최적화라기보다, **시작 시점에 잡히던 문제를 런타임으로 미루는 선택**에 가깝습니다.

## 무엇이 좋아질 수 있을까

- 시작 시간이 짧아질 수 있음
- 실제로 쓰지 않는 빈 초기화를 늦출 수 있음
- 개발 환경에서 가볍게 뜨는 느낌을 줄 수 있음

즉 "초기 부하를 뒤로 미루는 효과"는 분명 있습니다.

## 무엇이 위험해질까

### 1. 잘못된 의존성 문제가 늦게 터질 수 있다

원래는 애플리케이션 시작 중 바로 드러나던 설정 오류나 빈 생성 문제가, 실제 요청이 들어온 뒤에야 터질 수 있습니다.

### 2. 첫 요청이 느려질 수 있다

처음 호출되는 시점에 관련 빈이 한꺼번에 초기화되면 사용자는 첫 요청 지연으로 체감하게 됩니다.

### 3. 시작 성공이 정상 동작을 의미하지 않게 된다

로그상으로는 부팅 성공처럼 보여도, 실제 핵심 기능은 아직 검증되지 않은 상태일 수 있습니다.

## 특히 조심할 상황

아래 같은 서비스는 더 신중하게 보는 편이 좋습니다.

- 외부 API 클라이언트 빈이 많은 경우
- 설정 바인딩이 복잡한 경우
- 초기화 로직이 무거운 경우
- 운영에서 첫 요청 지연이 민감한 경우

## 실무 기준

전체 기본값으로 켜는 것보다, 정말 필요한 상황인지 먼저 따져보는 편이 낫습니다.

예를 들어

- 개발 환경에서만 시작 속도를 줄이고 싶은 경우
- 일부 보조 기능 빈만 늦춰도 충분한 경우

처럼 범위를 좁혀서 보는 편이 안전합니다.

## 정리

- Lazy Initialization은 공짜 성능 최적화가 아니다.
- 시작은 빨라질 수 있지만 문제 발견은 늦어질 수 있다.
- 첫 요청 지연과 런타임 오류 시점을 같이 감수할 수 있는지 먼저 봐야 한다.
- 전체 기본값으로 켜기보다 필요한 곳에 신중하게 적용하는 편이 보통 더 낫다.
