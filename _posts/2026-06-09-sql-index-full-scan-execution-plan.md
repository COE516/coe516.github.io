## 문제 상황

개발하다 보면 한 번쯤 이런 상황을 만납니다.

> 느려서 인덱스를 추가했는데, 왜 더 느려졌지?

많은 개발자가 인덱스를 성능을 높여주는 버튼처럼 생각합니다. 하지만 인덱스는 쿼리를 무조건 빠르게 만들어주는 기능이 아닙니다.

오히려 잘못 설계한 인덱스는 아래 문제를 만들 수 있습니다.

- 조회 성능 저하
- `INSERT`, `UPDATE`, `DELETE` 성능 저하
- 디스크 사용량 증가
- Optimizer의 잘못된 실행계획 선택

중요한 것은 "인덱스를 만들었다"가 아닙니다. **왜 이 인덱스를 만들었는지 설명할 수 있어야 합니다.**

## 인덱스에 대한 흔한 오해

### 인덱스는 많을수록 좋다?

아닙니다.

인덱스는 조회 성능을 높일 수 있지만, 데이터 변경 비용도 함께 증가시킵니다. 예를 들어 아래 쿼리가 실행된다고 해보겠습니다.

```sql
insert into employee
values (...);
```

DB는 테이블에만 데이터를 저장하지 않습니다.

```text
1. 테이블에 데이터 저장
2. 관련 인덱스 갱신
3. 통계 정보 반영
```

즉, 인덱스가 많을수록 쓰기 작업은 느려질 수 있습니다. 특히 변경이 잦은 테이블에 사용하지 않는 인덱스가 많이 붙어 있으면 조회보다 쓰기 비용에서 먼저 문제가 드러납니다.

### 인덱스를 만들면 무조건 사용된다?

이것도 아닙니다.

DB Optimizer는 인덱스를 사용하는 것이 비효율적이라고 판단하면 인덱스를 무시하고 Full Scan을 선택할 수 있습니다.

예를 들어 아래 쿼리를 보겠습니다.

```sql
select *
from employee
where status = 'Y';
```

전체 100만 건 중 95만 건이 `status = 'Y'`라면 어떨까요?

```text
전체 데이터: 1,000,000건
status = 'Y': 950,000건
```

이 경우 DB는 인덱스를 타고 95만 건을 찾는 것보다 테이블 전체를 읽는 편이 더 낫다고 판단할 수 있습니다. 인덱스는 "소수의 데이터를 빠르게 찾는 구조"에 가깝기 때문입니다.

## Full Scan과 Index Scan 차이

### Full Table Scan

Full Table Scan은 테이블 전체를 처음부터 끝까지 읽는 방식입니다.

```sql
select *
from employee
where status = 'Y';
```

동작 방식은 단순합니다.

```text
테이블의 첫 행부터 마지막 행까지 모두 읽는다.
```

Full Table Scan은 아래 상황에서 자주 발생합니다.

- 데이터 대부분을 조회하는 경우
- 인덱스 효율이 낮은 경우
- 조건의 선택도(selectivity)가 낮은 경우
- 통계 정보상 Full Scan 비용이 더 낮다고 판단되는 경우

선택도가 낮다는 것은 조건으로 걸러지는 데이터가 적다는 뜻입니다. 예를 들어 `성별`, `상태값`, `사용 여부`처럼 값의 종류가 적고 특정 값에 데이터가 몰려 있는 컬럼은 선택도가 낮은 경우가 많습니다.

### Index Scan 또는 Index Seek

Index Scan 또는 Index Seek는 인덱스를 통해 필요한 데이터를 찾는 방식입니다. DBMS마다 실행계획에 표시되는 용어는 조금씩 다르지만, 핵심은 인덱스를 통해 탐색 범위를 줄인다는 점입니다.

```sql
select *
from employee
where emp_id = 'A10001';
```

아래 조건에서는 인덱스 효율이 좋습니다.

- PK 조회
- 고유값 검색
- 결과 건수가 적은 조건
- 선택도가 높은 컬럼 조회

예를 들어 사번, 주문번호, 회원 ID처럼 특정 값으로 조회했을 때 결과 건수가 적은 컬럼은 인덱스 효과가 큽니다.

## 복합 인덱스 순서가 중요한 이유

실무에서 자주 놓치는 부분 중 하나가 복합 인덱스의 컬럼 순서입니다.

아래 인덱스가 있다고 해보겠습니다.

```sql
create index idx_employee_01
on employee (
    cmp_id,
    dept_id,
    user_name
);
```

이 인덱스의 컬럼 순서는 다음과 같습니다.

```text
cmp_id -> dept_id -> user_name
```

### 인덱스를 잘 사용하는 경우

```sql
select *
from employee
where cmp_id = 'COMP01';
```

```sql
select *
from employee
where cmp_id = 'COMP01'
  and dept_id = 'DEV';
```

위 쿼리는 복합 인덱스의 앞쪽 컬럼부터 조건에 사용하고 있기 때문에 인덱스를 효율적으로 사용할 가능성이 높습니다.

### 인덱스를 잘 사용하지 못하는 경우

```sql
select *
from employee
where user_name = '홍길동';
```

```sql
select *
from employee
where dept_id = 'DEV';
```

복합 인덱스는 기본적으로 왼쪽 컬럼부터 순서대로 탐색합니다. 이를 흔히 Leading Column Rule이라고 합니다.

즉, 아래 인덱스가 있을 때

```text
(cmp_id, dept_id, user_name)
```

앞쪽 컬럼인 `cmp_id`를 건너뛰고 `dept_id`나 `user_name`만으로 조회하면 인덱스 효율이 크게 떨어질 수 있습니다.

그래서 복합 인덱스를 만들 때는 단순히 WHERE 절에 등장하는 컬럼을 모두 넣는 것이 아니라, **어떤 조건 조합으로 가장 자주 조회하는지**를 먼저 봐야 합니다.

## WHERE 조건과 인덱스의 관계

인덱스가 있어도 WHERE 조건을 어떻게 작성하느냐에 따라 인덱스를 제대로 사용하지 못할 수 있습니다.

### 1. 컬럼에 함수 사용

```sql
where upper(user_name) = 'JY'
```

컬럼에 함수를 적용하면 DB는 컬럼 값을 먼저 계산해야 합니다. 이 경우 일반 인덱스를 그대로 활용하기 어렵습니다.

가능하면 저장 값을 정규화하거나, 조건 값을 컬럼에 맞춰 변환하는 방식이 좋습니다. 필요하다면 DBMS가 지원하는 함수 기반 인덱스도 검토할 수 있습니다.

### 2. 앞쪽에 `%`가 있는 LIKE

```sql
where user_name like '%김%'
```

앞쪽에 `%`가 있으면 인덱스가 탐색을 시작할 위치를 찾기 어렵습니다.

반면 아래 조건은 상대적으로 인덱스를 활용하기 쉽습니다.

```sql
where user_name like '김%'
```

문자열 포함 검색이 핵심 요구사항이라면 일반 B-Tree 인덱스만으로 해결하려고 하기보다 검색 전용 인덱스나 별도 검색 엔진을 검토하는 편이 나을 수 있습니다.

### 3. 컬럼 연산

```sql
where salary * 12 > 50000000
```

컬럼에 연산을 걸면 인덱스를 효율적으로 사용하기 어렵습니다. 가능하면 컬럼 자체는 그대로 두고 비교값을 변환하는 방식이 좋습니다.

```sql
where salary > 4166666
```

조건의 의미는 같더라도 DB가 인덱스를 사용할 수 있는 형태로 작성하는 것이 중요합니다.

### 4. 타입 불일치

```sql
where emp_id = 1001
```

만약 `emp_id` 컬럼이 문자열 타입이라면 내부적으로 타입 변환이 발생할 수 있습니다.

이 경우 아래처럼 컬럼 타입에 맞게 조건 값을 작성하는 편이 좋습니다.

```sql
where emp_id = '1001'
```

타입 변환이 컬럼 쪽에 걸리면 인덱스 사용이 어려워질 수 있습니다. 특히 운영에서는 작은 타입 불일치 하나가 예상과 다른 실행계획으로 이어질 수 있습니다.

## 인덱스를 타지 못하는 대표 사례

실무에서 자주 보는 패턴은 아래와 같습니다.

- 컬럼에 함수 사용
- 앞쪽 `%` LIKE
- `OR` 조건 남발
- 컬럼 계산
- 데이터 분포가 한쪽으로 몰린 경우
- 복합 인덱스의 컬럼 순서 미준수
- 조건 값과 컬럼의 타입 불일치

특히 `OR` 조건은 실행계획이 복잡해지는 경우가 많습니다.

```sql
select *
from employee
where emp_id = ?
   or user_name = ?;
```

상황에 따라서는 아래처럼 쿼리를 분리하는 방식이 더 나을 수 있습니다.

```sql
select *
from employee
where emp_id = ?

union all

select *
from employee
where user_name = ?;
```

물론 항상 `UNION ALL`이 정답은 아닙니다. 실제 데이터 분포와 실행계획을 확인한 뒤 판단해야 합니다.

## 운영에서는 실행계획을 먼저 본다

성능 이슈가 발생했을 때 감으로 판단하면 안 됩니다. 가장 먼저 확인해야 할 것은 실행계획입니다.

실행계획에서는 주로 아래 항목을 봅니다.

- Full Scan 발생 여부
- Index Seek 또는 Index Scan 여부
- Key Lookup 발생 여부
- Join 방식
- Cost가 높은 구간
- 예상 Rows와 실제 Rows 차이

예를 들어 실행계획에서 아래처럼 접근 방식을 확인할 수 있습니다.

```text
Index Seek
-> 인덱스를 통해 필요한 범위를 효율적으로 탐색

Table Scan
-> 테이블 전체를 읽고 있으므로 튜닝 검토 필요
```

다만 `Index Scan`이라고 해서 무조건 나쁘고, `Table Scan`이라고 해서 무조건 잘못된 것은 아닙니다. 데이터 대부분을 읽어야 하는 쿼리라면 Full Scan이 더 합리적일 수 있습니다.

중요한 것은 실행계획 이름 하나가 아니라 **왜 그 계획이 선택되었는지**입니다.

특히 개발 환경과 운영 환경은 아래 요소가 다를 수 있습니다.

- 데이터량
- 데이터 분포
- 통계 정보
- 실행계획

그래서 인덱스 튜닝은 가능하면 운영 데이터 기준으로 확인해야 합니다. 개발 DB에서 빠른 쿼리가 운영 DB에서도 빠르다는 보장은 없습니다.

## 실무 인덱스 설계 기준

인덱스를 설계할 때는 보통 아래 순서로 확인합니다.

```text
1. WHERE 절에 자주 사용되는 컬럼인가?
2. JOIN 조건에 사용되는 컬럼인가?
3. ORDER BY 또는 GROUP BY에 사용되는 컬럼인가?
4. 선택도가 높은 컬럼인가?
5. 데이터 분포가 한쪽으로 치우쳐 있지 않은가?
6. 실제 운영 데이터 기준으로 검증했는가?
```

그리고 복합 인덱스라면 추가로 아래를 봅니다.

- 자주 함께 조회되는 조건 조합인가?
- 앞쪽 컬럼부터 조건에 사용되는가?
- 동등 조건과 범위 조건의 순서가 적절한가?
- 기존 인덱스와 중복되지 않는가?

가장 중요한 기준은 단순합니다.

> 인덱스는 많이 만드는 것이 아니라 정확하게 만드는 것입니다.

## 정리

인덱스는 잘 사용하면 몇 초 걸리던 쿼리를 몇 ms 단위로 줄여줄 수 있습니다. 하지만 잘못 설계하면 오히려 조회 성능 저하, 불필요한 Scan, 쓰기 성능 악화, 디스크 사용량 증가를 유발할 수 있습니다.

그래서 실무에서는 아래 질문에 답할 수 있어야 합니다.

- 왜 이 인덱스를 만들었는가?
- 이 쿼리가 실제로 이 인덱스를 사용하는가?
- 운영 데이터에서도 효과가 있는가?
- 기존 인덱스와 중복되지는 않는가?

인덱스 튜닝은 감으로 하는 작업이 아닙니다. 쿼리, 데이터 분포, 실행계획을 함께 보고 판단해야 합니다.
