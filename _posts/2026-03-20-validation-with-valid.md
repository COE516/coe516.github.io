## 문제 상황

`@Valid`를 붙여두면 검증이 끝난 것처럼 느껴질 때가 있습니다. 하지만 실무에서 문제는 "검증을 했느냐"보다 **어떤 검증을 어디에 둬야 하느냐**에서 더 자주 생깁니다.

예를 들어 요청 형식 오류와 비즈니스 규칙 위반을 한 군데에 섞어버리면 책임이 금방 흐려집니다.

## `@Valid`가 잘하는 일

`@Valid`는 요청 형식 검증에 잘 맞습니다.

- 필수값 검증
- 문자열 길이 검증
- 이메일, 숫자 같은 형식 검증
- 중첩 요청 객체 기본 검증

```java
public record MemberCreateRequest(
    @NotBlank String name,
    @Email String email
) {
}
```

즉 DTO 수준에서 바로 걸러낼 수 있는 입력 오류를 빠르게 막는 데 유용합니다.

## `@Valid`만으로 해결되지 않는 것

아래는 보통 서비스 계층에서 보는 쪽이 자연스럽습니다.

- 이미 존재하는 이메일인지
- 현재 주문 상태에서 취소 가능한지
- 이 사용자가 수정 권한이 있는지
- 재고가 실제로 충분한지

이런 규칙은 요청 형식보다 도메인 상태와 데이터 조회에 더 가깝습니다.

## 실무에서 자주 꼬이는 부분

### DTO에 비즈니스 규칙을 과하게 넣는 경우

처음에는 한곳에서 검증해서 편해 보여도, DB 조회가 필요하거나 상황에 따라 달라지는 규칙은 DTO 검증에 억지로 넣기 시작하면 오히려 복잡해집니다.

### 서비스까지 단순 null 체크를 반복하는 경우

반대로 DTO에서 걸러도 되는 형식 오류를 서비스에서 또 검사하면 계층 책임이 겹칩니다.

### 에러 응답 형식을 계층마다 따로 만드는 경우

검증 실패 응답은 `@ControllerAdvice`로 한 번에 맞추는 편이 운영에서 훨씬 읽기 쉽습니다.

## 짧은 기준

- 형식 검증은 DTO + `@Valid`
- 상태 기반 검증은 Service
- 권한, 중복, 재고 같은 규칙은 비즈니스 검증으로 본다.
- 검증 실패 응답 형식은 전역에서 통일한다.

## 정리

- `@Valid`는 요청 형식 검증에 강하다.
- 비즈니스 규칙까지 대신해주지는 않는다.
- 중요한 것은 검증을 많이 넣는 것보다 책임을 맞는 계층에 두는 것이다.
- 형식 검증과 도메인 검증을 분리하면 코드가 훨씬 덜 꼬인다.
