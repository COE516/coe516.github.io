## 문제 상황

처음에는 콘솔 로그만으로도 충분해 보이지만, 운영에 들어가면 요청 추적, 에러 확인, 검색 편의성 때문에 로그 구성이 중요해집니다.

## 실무에서 자주 하는 실수

- 모든 로그를 DEBUG로 남김
- 콘솔과 파일 로그를 구분하지 않음
- 패턴에 요청 식별자 없음
- 운영 로그 롤링 정책 없음

이렇게 두면 로그가 많아질수록 읽기 어려워집니다.

## 기본 기준

실무에서는 보통 아래를 먼저 나눕니다.

- 로컬: 콘솔 위주
- 운영: 파일 또는 수집 시스템 위주
- 로그 레벨: 기본 INFO, 필요한 패키지만 DEBUG

## 예시

```xml
<springProfile name="prod">
    <root level="INFO">
        <appender-ref ref="FILE" />
    </root>
</springProfile>
```

프로파일별로 출력을 나누면 개발용 로그와 운영 로그가 섞이지 않습니다.

## 꼭 챙길 것

### 1. 롤링 정책

하루 단위 또는 용량 단위로 끊어야 합니다.

### 2. 요청 식별자

MDC를 써서 trace id나 request id를 남기면 장애 추적이 쉬워집니다.

### 3. 패키지별 레벨 조정

전체 DEBUG보다 필요한 패키지만 올리는 편이 훨씬 낫습니다.

## 정리

로그 설정은 많이 남기는 것보다 **읽히게 남기는 것**이 중요합니다.

핵심은 이렇습니다.

- 로컬과 운영 로그를 분리한다.
- 롤링 정책을 둔다.
- 요청 식별자를 남긴다.
- 필요한 패키지만 DEBUG로 조정한다.
