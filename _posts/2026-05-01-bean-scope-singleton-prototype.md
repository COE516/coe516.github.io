## 문제 상황

스프링 빈을 만들다 보면 "이 객체는 요청마다 새로 생기나", "왜 값이 공유되지" 같은 의문이 생깁니다. 특히 회원별 값이나 요청별 상태를 빈 필드에 잠깐 넣어두기 시작하면 singleton이 문제처럼 보일 때가 많습니다.

그런데 실무에서는 scope 문법보다 **상태를 빈 안에 두고 있느냐**가 더 핵심인 경우가 많습니다.

## 기본값은 singleton

스프링 빈은 기본적으로 singleton입니다. 컨테이너 안에서 하나를 만들고 계속 재사용합니다.

이 방식이 기본인 이유는 대부분의 서비스, 리포지토리, 클라이언트 빈이 상태 없이 동작하도록 만드는 편이 가장 단순하고 안정적이기 때문입니다.

## singleton에서 자주 생기는 착각

문제는 singleton 자체보다, singleton 빈 안에 요청별 상태를 넣는 순간 시작됩니다.

```java
@Service
public class CouponService {
    private Long currentMemberId;

    public void issue(Long memberId) {
        currentMemberId = memberId;
        // 쿠폰 발급 로직
    }
}
```

이 구조는 요청이 겹치면 값이 섞일 수 있습니다. 즉 "singleton이라서 문제"라기보다, **공유 객체에 요청별 상태를 넣은 설계**가 문제입니다.

그래서 scope를 바꾸기 전에 먼저 "이 값이 정말 필드에 있어야 하나"를 보는 편이 맞습니다.

## prototype은 무엇이 다를까

prototype은 요청할 때마다 새 인스턴스를 만듭니다.

겉으로 보면 "매번 새 객체가 필요할 때 쓰면 되겠네" 싶지만, 여기서 자주 놓치는 부분이 있습니다.

### singleton 안에 prototype을 그냥 주입하면

주입 시점에 받은 한 인스턴스를 계속 쓰게 됩니다.

즉 아래처럼 생각하면 틀릴 수 있습니다.

- prototype 빈을 주입했으니 메서드 호출마다 새 객체겠지
- singleton 서비스 안에서도 자동으로 새로 만들어지겠지

실제로 매번 새 객체가 필요하면 `ObjectProvider` 같은 조회 방식이 필요할 수 있습니다.

## 언제 prototype을 검토할까

아래처럼 정말 독립 인스턴스가 반복해서 필요할 때는 검토할 만합니다.

- 매번 새 작업 컨텍스트 객체가 필요함
- 생성 비용은 작지만 내부 상태를 잠깐 들고 써야 함
- singleton으로 두면 상태 충돌이 분명함

다만 이 경우에도 먼저 메서드 지역 변수나 일반 객체 생성으로 충분하지 않은지 보는 편이 낫습니다.

## 실무에서는 보통 이렇게 본다

### 상태 없는 빈이면 singleton

대부분의 서비스 빈은 여기입니다. 가장 무난하고 운영하기 쉽습니다.

### 요청마다 다른 값이면 필드보다 메서드 인자

사용자 ID, 요청 옵션, 임시 계산값은 빈 상태보다 메서드 인자나 지역 변수로 흘리는 편이 훨씬 안전합니다.

### 그래도 독립 객체가 필요하면 prototype보다 구조를 먼저 본다

정말 스프링 빈이어야 하는지, 그냥 `new`로 만드는 일반 객체가 더 단순한지 먼저 확인합니다.

## 짧은 기준

- 기본은 singleton으로 본다.
- scope보다 상태 공유 설계가 더 중요하다.
- 요청별 데이터는 빈 필드보다 메서드 인자나 지역 변수로 둔다.
- prototype은 매번 새 인스턴스가 정말 필요할 때만 신중히 쓴다.

## 정리

- 스프링 빈의 기본값은 singleton이다.
- 많은 문제는 scope보다 공유 빈 안에 상태를 넣는 설계에서 시작된다.
- prototype은 자동 만능 해법이 아니고, 주입 방식까지 같이 봐야 한다.
- scope를 바꾸기 전에 정말 상태를 어디에 둘지부터 다시 보는 편이 더 안전하다.
