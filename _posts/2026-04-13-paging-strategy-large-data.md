## 문제 상황

처음에는 `page`, `size`만 있으면 충분해 보입니다. 하지만 데이터가 많아질수록 뒤쪽 페이지 응답이 점점 느려지거나, 페이지 이동은 되는데 쿼리 비용이 과하게 커지는 경우가 생깁니다.

이때 중요한 것은 "어떤 페이징이 더 고급인가"가 아니라 **화면 요구사항과 쿼리 비용 중 무엇을 우선해야 하느냐**입니다.

## offset 기반은 왜 편하고, 왜 느려질까

offset 기반은 가장 익숙합니다.

```sql
select *
from orders
order by id desc
limit 20 offset 10000;
```

장점은 분명합니다.

- 구현이 단순하다.
- 페이지 번호 UI와 잘 맞는다.
- 임의 페이지로 바로 이동하기 쉽다.

대신 뒤로 갈수록 DB가 앞부분을 더 많이 건너뛰어야 해서 비용이 커질 수 있습니다. 큰 테이블에서는 페이지 번호는 단순한데 실제 쿼리는 점점 무거워지는 식입니다.

## keyset 기반은 언제 유리할까

keyset 방식은 마지막으로 본 기준값 다음부터 가져오는 방식입니다.

```sql
select *
from orders
where id < 105230
order by id desc
limit 20;
```

이 방식은 보통 아래 상황에서 강합니다.

- 무한 스크롤
- 최신순 피드
- 계속 다음 페이지로만 넘기는 조회
- 데이터가 아주 큰 테이블

앞부분을 건너뛰지 않기 때문에 뒤로 갈수록 느려지는 문제가 훨씬 덜합니다.

## keyset에서 자주 놓치는 부분

좋은 점만 있는 것은 아닙니다.

- 페이지 번호 UI와 잘 맞지 않는다.
- "123페이지로 바로 가기"가 어렵다.
- 정렬 기준이 명확해야 한다.
- 동점 정렬이 있으면 기준 컬럼을 더 신중히 잡아야 한다.

예를 들어 `created_at`만으로 정렬하면 같은 시각 데이터에서 누락이나 중복이 생길 수 있어, 보통 `created_at`, `id` 같이 묶어서 기준을 잡습니다.

## 보통 이렇게 고른다

### offset이 잘 맞는 경우

- 관리자 화면
- 전체 개수와 페이지 번호가 중요한 화면
- 데이터 규모가 아직 크지 않음
- 임의 페이지 이동 요구가 큼

### keyset이 잘 맞는 경우

- 무한 스크롤
- 최근 데이터 위주 조회
- 대용량 테이블
- 성능이 페이지 번호 UX보다 더 중요함

## 선택 전에 같이 볼 것

전략만 바꾸기 전에 아래도 함께 봐야 합니다.

- 정렬 컬럼에 인덱스가 맞는가
- 화면이 정말 페이지 번호 이동이 필요한가
- 전체 개수 count 쿼리 비용이 큰가
- 중복/누락 없이 이어 읽을 기준 컬럼이 있는가

## 정리

- offset은 구현과 UX가 단순하지만, 뒤쪽 페이지로 갈수록 비용이 커질 수 있다.
- keyset은 대용량과 무한 스크롤에 유리하지만, 페이지 번호 기반 UI에는 불편하다.
- 어떤 전략이 더 고급이냐보다 화면 요구사항과 정렬 기준에 맞는지가 더 중요하다.
- 대용량 목록이 느려지면 offset 크기, count 비용, 정렬 인덱스부터 먼저 같이 확인하는 편이 좋다.
