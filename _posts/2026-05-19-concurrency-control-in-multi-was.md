## 문제는 멀티 WAS 전환 이후에 보였다

실무에서 단일 WAS 환경에서는 별문제 없이 운영되던 기능이 있었습니다.

전자결재 완료 시 외부 시스템으로 상태를 연동하는 기능이었는데, 내부 결재 상태가 완료로 바뀌면 WSDL/SOAP 호출을 보내 외부 시스템 상태도 함께 변경하는 구조였습니다.

흐름은 단순했습니다.

```text
결재 완료
   ↓
GW WAS
   ↓
WSDL/SOAP 호출
   ↓
외부 시스템 상태 변경
```

초기에는 단일 서버 환경이었기 때문에 눈에 띄는 문제 없이 운영됐습니다.

그런데 WAS가 이중화되면서 같은 결재 건이 중복 연동되는 문제가 발생했습니다.

이번 글은 그때 겪었던 문제를 기준으로, 멀티 WAS 환경에서 왜 `synchronized`가 충분하지 않았는지와 어떤 방식으로 구조를 바꿨는지 정리한 내용입니다.

## 1. 단일 서버에서는 왜 괜찮아 보였을까

기존 로직은 전자결재 상태 변경 시 외부 시스템으로 WSDL 호출을 수행하는 구조였습니다.

단일 WAS 환경에서는 요청이 하나의 JVM 내부에서 처리되기 때문에, 같은 시점에 같은 로직이 부딪혀도 결국 한 프로세스 안에서만 경쟁이 일어났습니다.

일부 로직은 아래처럼 `synchronized`로 감싸 두기도 했습니다.

```java
synchronized (this) {
    callWsdl();
}
```

당시에는 이 정도면 충분해 보였습니다.

- 요청이 같은 애플리케이션 프로세스로 들어온다.
- lock을 보는 주체도 같은 JVM이다.
- 동시 실행 범위가 서버 한 대 안으로 제한된다.

즉, 코드가 분산 환경을 잘 처리해서가 아니라 **운영 환경이 단순해서 문제가 잘 드러나지 않았던 것**에 가까웠습니다.

## 2. 서버가 2대가 되자 문제가 시작됐다

WAS를 2대로 늘린 이후부터 간헐적으로 이상 현상이 보이기 시작했습니다.

운영 로그에서 주로 확인된 증상은 아래와 같았습니다.

- 동일 결재 건 WSDL 중복 호출
- 동일 상태값 재전송
- lock 재시도 횟수 초과
- 대기 스레드 누적

특히 결재 완료 건이 한꺼번에 몰릴 때 문제가 더 잘 드러났습니다.

거의 동시에 동일 건이 여러 서버에서 처리되면 구조는 아래처럼 됩니다.

```text
Load Balancer
    ↓
    ├─ WAS1 → 완료 연동 시작
    └─ WAS2 → 완료 연동 시작
```

결과적으로 같은 결재 건을 서로 다른 WAS가 동시에 처리하려고 시도했습니다.

단일 서버 시절에는 잘 안 보이던 문제가, 멀티 WAS 환경에서는 실제 운영 장애로 드러난 것입니다.

## 3. 왜 `synchronized`가 안 먹혔을까

처음에는 가장 단순한 의문이 들었습니다.

> `synchronized`를 걸어놨는데 왜 중복 실행될까?

답은 생각보다 단순했습니다.

`synchronized`는 현재 JVM 안에서만 유효한 lock입니다.

서버가 2대라면 실제로는 아래와 같습니다.

```text
WAS1 = JVM #1
WAS2 = JVM #2
```

WAS1에서 lock을 잡아도 WAS2는 그 사실을 알지 못합니다.

```java
synchronized (this) {
    callWsdl();
}
```

이 코드는 단일 서버에서는 의미가 있지만, 멀티 WAS에서는 각 서버가 **자기 JVM 안에서만** 잠그는 셈입니다.

그래서 정리하면 이렇게 이해하는 편이 맞았습니다.

- `synchronized`는 단일 JVM 동시성 제어에는 유효할 수 있다.
- 여러 WAS 인스턴스가 동시에 처리하는 문제는 막아주지 못한다.

이 차이를 실제 장애를 겪고 나서 더 분명하게 체감했습니다.

## 4. 처음 적용했던 방식과 한계

중복 연동을 막기 위해 처음에는 분산 lock에 가까운 방식을 적용했습니다.

의도는 명확했습니다.

- 결재 완료 이벤트가 들어오면 lock 획득 시도
- lock 획득에 성공한 서버만 WSDL 호출 수행
- 실패한 쪽은 재시도

흐름만 단순화하면 아래와 비슷했습니다.

```text
결재 완료
   ↓
lock 획득 시도
   ├─ 성공 → WSDL 호출
   └─ 실패 → 재시도
```

초기에는 어느 정도 효과가 있었지만, 운영 중 다른 문제가 나타났습니다.

대량 결재 시 lock 경쟁이 심해졌고 아래와 같은 현상이 발생했습니다.

- lock 재시도 횟수 초과 예외 발생
- 대기 중인 스레드 증가
- 특정 시간대 처리량 저하

즉, 동시 실행 자체는 줄였지만 이번에는 **lock 경쟁과 대기 비용**이 병목이 됐습니다.

정리하면 이 단계의 문제는 이것이었습니다.

> 동시성은 어느 정도 막았지만, 처리량과 안정성이 함께 나빠졌다.

## 5. 실제로 적용한 해결 방법

결국 구조를 조금 바꿨습니다.

핵심은 단순했습니다.

> 동시에 실행하지 말고, 순차 처리하자.

### Queue 기반 처리로 변경

결재 완료 시점에 바로 WSDL 호출을 수행하지 않고, 먼저 Queue에 적재한 뒤 별도 처리 흐름에서 순차적으로 꺼내 실행하도록 바꿨습니다.

구조는 아래와 비슷합니다.

```text
결재 완료
   ↓
Queue 적재
   ↓
Dispatcher Thread
   ↓
WSDL 호출
```

여기서 중요한 포인트는 같은 종류의 작업이 한 줄로 서서 처리되도록 만든 것입니다.

예를 들어 `lockName` 기준으로 Queue를 나눠 아래처럼 구성했습니다.

- `QUEUE_A`
- `QUEUE_B`
- `QUEUE_C`

그리고 같은 종류의 작업은 FIFO 방식으로 순차 처리하도록 맞췄습니다.

이렇게 바꾸고 나서 얻은 효과는 비교적 명확했습니다.

- 동일 결재 건 중복 호출 감소
- WAS 간 동시 처리 충돌 완화
- 대량 결재 시 처리 안정성 향상

완전히 모든 문제가 자동으로 사라지는 것은 아니지만, 적어도 lock 재시도와 스레드 대기로 버티던 구조보다 운영 안정성이 더 좋아졌습니다.

## 6. 운영하면서 느낀 점

멀티 WAS 환경에서는 아래 문장이 가장 위험할 수 있습니다.

> 단일 서버에서는 문제 없었다.

이 말은 실제로는 안전하다는 뜻이 아니라, **아직 분산 환경에서 검증되지 않았다**는 뜻일 때가 많았습니다.

특히 아래 같은 로직은 멀티 WAS 전환 시 다시 보는 편이 좋았습니다.

- 배치
- 외부 연동
- 정산 처리
- 상태 변경
- 중복 생성 가능 로직

지금은 기능을 볼 때 먼저 이 질문을 하게 됩니다.

> 이 로직은 WAS가 2대여도 안전할까?

이 질문 하나만 해도 단일 JVM 기준 방어로 충분한지, 아니면 queue·공통 저장소·분산 제어 기준으로 다시 설계해야 하는지 더 빨리 판단할 수 있었습니다.

## 정리

- 단일 WAS에서 멀쩡하던 로직도 멀티 WAS에서는 쉽게 깨질 수 있습니다.
- `synchronized`는 JVM 내부 lock이라 여러 서버 사이의 중복 실행을 막지 못합니다.
- 분산 lock은 문제를 줄일 수 있지만, 경쟁이 심해지면 재시도와 대기 비용이 병목이 될 수 있습니다.
- 외부 연동처럼 같은 작업을 동시에 처리할 필요가 없는 경우에는 queue 기반 순차 처리가 더 현실적인 선택이 될 수 있습니다.
- 멀티 WAS 환경에서는 구현보다 먼저 "이 로직이 여러 서버에서 동시에 실행돼도 안전한가"를 확인하는 습관이 중요합니다.
