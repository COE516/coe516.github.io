## 단일 서버에서는 왜 문제가 잘 안 보일까

단일 WAS 환경에서는 동시에 같은 로직이 들어오더라도 결국 **같은 JVM 안에서** 부딪히는 경우가 많습니다.

그래서 아래 같은 방식이 어느 정도는 통했습니다.

```java
synchronized (this) {
    process();
}
```

또는 상태값 하나로 재진입을 막기도 했습니다.

```java
if (!isRunning) {
    isRunning = true;
    process();
}
```

이 방식이 단일 서버에서 버텼던 이유는 간단합니다.

- 요청이 같은 프로세스 안으로 들어온다.
- lock과 상태값을 같은 메모리에서 본다.
- 동시에 실행되는 범위가 JVM 내부로 제한된다.

즉 코드가 완벽해서가 아니라, **환경이 단순해서 문제가 덜 드러난 것**에 가까웠습니다.

## WAS가 2대가 되면 왜 바로 터질까

WAS를 2대로 늘리면 같은 요청이 서로 다른 서버로 분산될 수 있습니다.

```text
사용자 요청
   ↓
Load Balancer
   ├─ WAS A → 처리 시작
   └─ WAS B → 처리 시작
```

이 순간부터는 "한 번만 실행돼야 하는 로직"이 실제로는 두 번 실행될 수 있습니다.

대표적으로 문제가 나는 구간은 아래처럼 비슷했습니다.

- 특정 API는 한 번만 처리돼야 한다.
- 배치는 동시에 하나만 돌아야 한다.
- 결재나 정산 데이터는 중복 생성되면 안 된다.
- 외부 API 호출은 한 번만 나가야 한다.

단일 WAS에서는 괜찮았던 코드가 멀티 WAS로 넘어오면 갑자기 운영 이슈가 되는 이유가 여기 있습니다.

## `synchronized`가 안 먹히는 이유

멀티 WAS 환경에서 가장 자주 나오는 질문이 이것입니다.

> `synchronized`를 걸었는데 왜 중복 실행될까?

핵심은 `synchronized`가 **현재 JVM 내부에서만 유효한 lock**이라는 점입니다.

서버가 2대라면 실제 구조는 아래에 가깝습니다.

```text
WAS A = JVM #1
WAS B = JVM #2
```

WAS A가 lock을 잡아도 WAS B는 그 사실을 모릅니다.

```java
synchronized (this) {
    process();
}
```

이 코드는 서버 A 안에서는 lock이지만, 서버 B 입장에서는 아무 제약이 없습니다.

그래서 이해를 이렇게 바꾸는 편이 맞았습니다.

- 단일 JVM 문제면 `synchronized`가 도움이 될 수 있다.
- 멀티 WAS 문제면 JVM 밖에서 기준을 잡아야 한다.

같은 서버 안에서도 인스턴스가 나뉘면 기대와 다르게 동작할 수 있지만, 멀티 WAS에서는 그 한계가 더 명확하게 드러납니다.

## 실제로 자주 터졌던 문제

### 1. 배치 중복 실행

스케줄러가 각 서버에서 동시에 돌면 같은 데이터를 두 번 처리할 수 있습니다.

```text
WAS A → 새벽 배치 시작
WAS B → 새벽 배치 시작
```

이 경우 보통 아래 문제가 따라왔습니다.

- 동일 데이터 중복 INSERT
- 동일 외부 API 중복 호출
- 상태값 꼬임

### 2. API 중복 처리

사용자 버튼 연타, 네트워크 재시도, 클라이언트 중복 호출이 겹치면 같은 요청이 여러 번 들어옵니다.

멀티 서버에서는 그 요청이 서로 다른 WAS로 분산될 수 있어서 단일 서버 때보다 훨씬 쉽게 중복 처리됩니다.

### 3. 결재·정산 데이터 중복 생성

실무에서는 이 케이스가 가장 복구 비용이 컸습니다.

예를 들면 아래처럼 이어집니다.

- 동일 전표가 두 건 생성됨
- 동일 결재 문서가 두 번 만들어짐
- 동일 정산 건이 중복 적재됨

이런 데이터는 코드 수정으로 끝나지 않고 운영에서 수작업 정리가 필요해지는 경우가 많습니다.

## 검토했던 해결 방법들

### 1. DB Lock

가장 먼저 떠올리기 쉬운 방식입니다.

```sql
SELECT *
FROM JOB_TABLE
WHERE ID = ?
FOR UPDATE
```

장점은 구현이 비교적 단순하고, DB 기준으로 동시성 제어를 걸 수 있다는 점입니다.

다만 아래 부담이 있습니다.

- DB 부하 증가
- deadlock 가능성
- 장시간 lock 유지 시 대기 증가

### 2. Redis 분산 Lock

멀티 WAS 환경에서 많이 검토하는 방식입니다.

- `SETNX`
- Redisson

장점은 여러 서버 간 락을 잡기 좋고 성능도 나쁘지 않다는 점입니다.

반대로 Redis 인프라가 없으면 도입 비용이 생기고, 운영 복잡도도 함께 올라갑니다.

### 3. Queue 기반 순차 처리

아예 동시에 처리하지 않고 순서를 강제하는 방식입니다.

- RabbitMQ
- Kafka

주문, 정산, 후처리처럼 순차성이 중요한 흐름에서는 꽤 강력합니다.

대신 구조 변경 폭이 크고, 기존 동기식 로직을 비동기 구조로 바꿔야 하는 부담이 있습니다.

### 4. 상태 플래그 방식

DB의 상태값을 이용해 먼저 선점하는 방식입니다.

예를 들면 이런 상태를 둡니다.

- `READY`
- `PROCESSING`
- `DONE`

처리 전에 상태를 먼저 바꾸고, 이미 다른 서버가 선점했으면 재진입하지 않는 구조입니다.

기존 구조를 크게 흔들지 않으면서 적용할 수 있어서 실무에서 자주 검토했습니다.

## 실제로 적용한 방법

운영에서는 **DB 기반 상태 플래그 + 중복 체크** 방식을 적용했습니다.

선택 이유는 명확했습니다.

- 기존 구조 변경을 최소화하고 싶었다.
- Redis 인프라가 없었다.
- 장애 추적이 DB 기준으로 더 쉬웠다.
- 운영 안정성을 우선해야 했다.

핵심은 처리 전에 상태를 원자적으로 바꾸는 것입니다.

```sql
UPDATE JOB_TABLE
SET STATUS = 'PROCESSING'
WHERE JOB_ID = ?
  AND STATUS = 'READY'
```

그리고 update 결과 건수로 실행 여부를 판단합니다.

- `1건` 업데이트 성공 → 내가 선점했으니 처리 진행
- `0건` 업데이트 성공 → 이미 다른 서버가 처리 중이므로 종료

이 방식의 장점은 판단 기준이 애플리케이션 메모리가 아니라 **공통 DB 상태**라는 점입니다.

즉 서버가 몇 대이든 결국 같은 row를 기준으로 경쟁하므로, 멀티 WAS 환경에서도 비교적 단순하게 제어할 수 있습니다.

## 적용할 때 같이 봐야 하는 예외 상황

상태 플래그 방식도 끝이 아닙니다. 운영에서는 실패 복구 전략까지 같이 있어야 안정적이었습니다.

### 1. `PROCESSING` 상태에서 장애가 나는 경우

서버가 처리 중에 죽으면 상태가 `PROCESSING`에 남을 수 있습니다.

이 경우 다음 실행이 영원히 막히지 않도록 재처리 기준이 필요합니다.

예를 들면 이런 보조 컬럼을 둡니다.

- `updated_at`
- `worker_id`
- `retry_count`

### 2. timeout 기준이 없는 경우

얼마나 오래 `PROCESSING`이면 비정상으로 볼지 정하지 않으면 운영자가 매번 수동 판단해야 합니다.

배치와 API는 허용 시간이 다르기 때문에 업무 특성에 맞는 기준이 필요했습니다.

### 3. 중복 생성 자체를 DB에서 막지 않은 경우

애플리케이션 제어만 믿으면 누락이 생길 수 있습니다.

정말 중복되면 안 되는 데이터라면 가능하면 아래도 같이 검토하는 편이 안전했습니다.

- unique key
- 중복 요청 식별 키
- 멱등성(idempotency) 기준 값

즉 애플리케이션 레벨 제어와 데이터 레벨 방어를 같이 두는 편이 운영에서 더 버텼습니다.

## 운영하면서 체크하게 된 질문

멀티 WAS 환경에서는 작은 기능도 다시 보게 됩니다.

특히 아래 로직은 단일 서버 기준으로 짜여 있지 않은지 점검할 필요가 있습니다.

- 배치
- 정산
- 결재 생성
- 외부 API 호출
- 상태 변경 로직

개인적으로는 구현 전에 이 질문을 먼저 하게 됐습니다.

> 이 로직이 서버 2대에서도 안전할까?

이 질문 하나만으로도 단일 JVM 기준 코드인지, DB나 분산 환경 기준으로 다시 설계해야 하는지 훨씬 빨리 구분할 수 있었습니다.

## 정리

- 단일 WAS에서 멀쩡하던 로직도 멀티 WAS에서는 쉽게 깨질 수 있습니다.
- `synchronized`는 JVM 내부 lock이라 여러 서버 사이를 막아주지 못합니다.
- 중복 실행이 치명적인 구간은 공통 저장소 기준으로 제어해야 합니다.
- 상태 플래그 방식은 구조 변경이 작고 추적이 쉬워 실무에서 현실적인 선택이 될 수 있습니다.
- 다만 `PROCESSING` 복구, timeout, unique key 같은 운영 장치까지 함께 설계해야 합니다.

서버를 늘리는 순간 성능만 좋아지는 것이 아니라 **동시성 조건도 달라진다**는 점을 먼저 의식하면, 멀티 WAS 전환 이후의 운영 장애를 꽤 많이 줄일 수 있습니다.
