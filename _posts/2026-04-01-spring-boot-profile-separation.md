## 문제 상황

로컬에서 편하게 쓰던 설정이 운영에도 그대로 들어가면 로그 레벨, DB 연결, 외부 URL, 캐시 정책까지 같이 꼬일 수 있습니다.

특히 처음에는 잘 돌아가다가도, 운영에서만 필요한 값과 개발 편의값이 섞이기 시작하면 어디를 바꿔야 하는지 금방 헷갈립니다.

## 기본 원칙

프로파일 분리는 단순히 파일을 나누는 작업이 아니라 **환경별로 달라져야 하는 값만 분리하는 작업**입니다.

보통은 이렇게 가져갑니다.

- `application.yml`: 공통값
- `application-dev.yml`: 로컬/개발값
- `application-prod.yml`: 운영값

핵심은 모든 값을 복사해 넣는 것이 아니라, 차이가 있는 값만 분리하는 것입니다.

## 어떤 값을 나누나

주로 아래 같은 값이 분리 대상입니다.

- datasource 정보
- 로그 레벨
- 외부 API 주소
- 캐시 사용 여부
- 배치/스케줄 활성화 여부

반대로 모든 값을 환경 파일에 중복으로 넣기 시작하면 관리가 더 어려워집니다.

## 실행 시 주의할 점

활성 profile을 명확히 해야 합니다.

```bash
--spring.profiles.active=prod
```

운영에서는 소스 파일보다 환경변수나 배포 설정으로 profile을 고정하는 경우가 많습니다.

## 실무에서 자주 꼬이는 부분

### 1. 공통값까지 환경 파일마다 복사하는 경우

처음에는 편해 보여도 나중에는 어떤 값이 기준인지 찾기 어려워집니다.

### 2. 운영 민감값을 yml에 같이 두는 경우

DB 비밀번호, API 키 같은 값은 profile 파일보다 외부 주입 방식이 더 안전합니다.

### 3. 실제 활성 profile을 확인하지 않는 경우

설정을 바꿨는데 반영이 이상하면, 파일 내용보다 먼저 지금 어떤 profile로 떴는지를 확인하는 편이 빠릅니다.

## 실무 팁

- 공통값은 기본 파일에 둔다.
- 환경 차이가 있는 값만 profile 파일로 보낸다.
- 비밀값은 yml보다 외부 주입을 우선한다.
- 문제를 볼 때는 항상 실제 활성 profile부터 확인한다.

## 정리

- 프로파일 분리는 파일 개수를 늘리는 작업이 아니다.
- 공통값과 환경값의 경계를 분명히 하는 작업에 가깝다.
- 운영 민감값은 외부 주입과 함께 가져가는 편이 안전하다.
- 설정이 이상하면 파일보다 먼저 실제 활성 profile과 최종 적용값을 확인하는 편이 빠르다.
