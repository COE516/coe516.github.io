## 문제 상황

트랜잭션이 간헐적으로 실패하고 DB 로그에 deadlock이 보이면 원인을 잡기 어렵습니다.

특히 재현이 잘 안 되면 코드가 문제인지, 트래픽 패턴이 문제인지, 쿼리 순서가 문제인지 판단이 더 어려워집니다.

## 데드락은 왜 생길까

핵심은 단순합니다. 서로 다른 트랜잭션이 자원을 하나씩 잡은 채, 상대가 가진 자원을 기다리면 사이클이 생깁니다.

예를 들어 이런 패턴입니다.

- 트랜잭션 A: row 1 잠금 후 row 2 대기
- 트랜잭션 B: row 2 잠금 후 row 1 대기

이 상태가 되면 DB는 둘 중 하나를 희생시켜 롤백합니다.

## 실무에서는 이 순서로 본다

### 1. 어떤 쿼리 조합에서 충돌하는지 본다

먼저 deadlock 로그나 DB 진단 정보를 봐야 합니다.

중요한 것은 "데드락이 났다"가 아니라 아래입니다.

- 어떤 테이블이었는가
- 어떤 인덱스를 탔는가
- 어떤 SQL 두 개가 서로 충돌했는가

애플리케이션 코드만 보고 추측하면 방향이 쉽게 빗나갑니다.

### 2. 잠금 순서가 일관적인지 본다

데드락의 가장 흔한 원인 중 하나는 같은 데이터를 서로 다른 순서로 잡는 것입니다.

예를 들어 어떤 로직은 주문 후 재고를 잡고, 다른 로직은 재고 후 주문을 잡으면 충돌 가능성이 커집니다.

그래서 같은 자원을 다루는 흐름은 **잠금 순서를 최대한 통일**하는 편이 중요합니다.

### 3. 트랜잭션이 너무 길지 않은지 본다

트랜잭션이 길어질수록 잠금을 오래 잡고 있게 됩니다.

특히 아래 패턴이 자주 문제를 만듭니다.

- 트랜잭션 안에서 외부 API 호출
- 한 번에 너무 많은 row 처리
- 사용자 입력 대기나 긴 비즈니스 로직 포함

잠금 시간을 줄이는 것만으로도 빈도가 크게 내려가는 경우가 많습니다.

## 자주 쓰는 대응

- 잠금 순서를 일정하게 맞춘다.
- 트랜잭션 범위를 줄인다.
- 한 번에 잡는 행 수를 줄인다.
- 재시도 가능한 구조를 둔다.

특히 데드락은 잘 설계해도 완전히 0으로 만들기 어려운 경우가 있어, 재시도 전략을 같이 두는 편이 현실적입니다.

## 정리

- 데드락은 원리보다 충돌한 SQL과 잠금 순서를 먼저 봐야 빨리 좁혀진다.
- 같은 자원을 다른 순서로 잡는 흐름은 가장 먼저 의심할 만하다.
- 긴 트랜잭션과 대량 잠금은 데드락 빈도를 키운다.
- 완전 제거보다 빈도 감소와 재시도 가능한 복구 구조를 함께 가져가는 편이 실무적이다.
