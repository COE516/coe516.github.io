## 문제 상황

로그인 체크, 로깅, 요청 추적 같은 공통 로직을 어디에 둘지 고민할 때 필터와 인터셉터가 자주 같이 나옵니다.

둘 다 요청 전후를 다루기 때문에 비슷해 보이지만, 실제로는 동작 위치와 접근 가능한 정보가 다릅니다. 그래서 이름보다 **무슨 책임을 맡길지** 기준으로 나누는 편이 덜 헷갈립니다.

## 가장 큰 차이

- 필터: 서블릿 레벨
- 인터셉터: 스프링 MVC 레벨

즉 필터가 더 앞단에서 동작합니다.

## 핵심 기준

헷갈릴 때는 이 세 가지를 먼저 보면 빠릅니다.

- 컨트롤러를 몰라도 되는 전역 처리인가
- `HandlerMethod` 같은 MVC 정보가 필요한가
- DispatcherServlet 앞단에서 처리해야 하는가

앞쪽이면 필터, 뒤쪽이면 인터셉터 쪽일 가능성이 큽니다.

## 비교해서 보면

### 필터

- 실행 위치: 서블릿 레벨
- 접근 가능한 정보: `HttpServletRequest`, `HttpServletResponse`
- 잘 맞는 책임: 전역 로깅, 인코딩, 공통 헤더 검사, 요청/응답 래핑

### 인터셉터

- 실행 위치: 스프링 MVC 레벨
- 접근 가능한 정보: 컨트롤러 진입 전후 흐름, `HandlerMethod`
- 잘 맞는 책임: 로그인 체크, 권한 확인, 컨트롤러 공통 처리

## 필터가 잘 맞는 경우

필터는 컨트롤러를 모르더라도 처리해야 하는 작업에 잘 맞습니다.

- 인코딩 설정
- 공통 로깅
- 인증 토큰의 1차 처리
- 요청/응답 래핑

예를 들어 모든 요청에 trace id를 심거나, 공통 헤더를 검사하는 일은 필터가 자연스럽습니다.

## 인터셉터가 잘 맞는 경우

인터셉터는 스프링 MVC 흐름 안에서 컨트롤러와 더 가깝게 동작합니다.

- 로그인 체크
- 권한 확인
- 컨트롤러 진입 전후 공통 처리
- `HandlerMethod` 정보 활용이 필요한 경우

즉 "어떤 컨트롤러를 호출하는지 알아야 하는 처리"에 더 잘 맞습니다.

## 예를 들면 이렇게 나눈다

- 모든 요청에 trace id를 부여한다 → Filter
- 특정 컨트롤러 진입 전 로그인 여부를 본다 → Interceptor
- 공통 헤더 존재 여부를 본다 → Filter
- 컨트롤러 메서드 기준으로 접근 정책을 나눈다 → Interceptor

## 자주 하는 실수

### 인증/인가를 전부 필터에 몰아넣는 경우

가능은 하지만, 컨트롤러별 예외 처리나 세밀한 접근 제어가 필요하면 구조가 금방 복잡해질 수 있습니다.

특히 스프링 MVC 정보가 필요한 순간부터는 인터셉터나 Spring Security 구조가 더 자연스러운 경우가 많습니다.

### 반대로 전역 로깅까지 인터셉터에 넣는 경우

MVC 바깥에서도 일관되게 처리해야 하는 작업이라면 필터가 더 자연스러울 수 있습니다.

예를 들어 정적 리소스, 예외 응답, 일부 비MVC 요청 흐름까지 같은 규칙으로 보고 싶다면 인터셉터만으로는 빠지는 구간이 생길 수 있습니다.

## 정리

- 서블릿 전역 처리면 필터가 잘 맞는다.
- 스프링 MVC 흐름 제어면 인터셉터가 잘 맞는다.
- 기준은 "컨트롤러 정보를 알아야 하는가"다.
- 이름보다 책임을 먼저 나누면 공통 로직 위치를 결정하기 쉬워진다.
