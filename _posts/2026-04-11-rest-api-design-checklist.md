## 문제 상황

API를 만들다 보면 URL 이름은 신경 쓰는데, 정작 상태 코드, 응답 구조, 오류 형식, 멱등성 같은 중요한 부분은 뒤늦게 맞추는 경우가 많습니다.

처음에는 작은 차이처럼 보여도 API 수가 늘어나면 클라이언트는 매번 다른 규칙을 기억해야 하고, 운영 중 오류 분석도 더 어려워집니다. 그래서 REST API 설계는 예쁜 URL보다 **어떤 규칙을 끝까지 동일하게 가져갈 것인가**가 더 중요합니다.

## 무엇부터 먼저 맞춰야 할까

실무에서는 보통 아래 순서로 보는 편이 덜 흔들립니다.

1. 자원과 행위를 어떻게 나눌지
2. 성공/실패 시 상태 코드를 어떻게 줄지
3. 응답 본문 구조를 어떤 기준으로 통일할지
4. 목록, 페이징, 정렬 규칙을 어떻게 맞출지
5. 재시도 가능한 요청과 아닌 요청을 어떻게 구분할지

즉 URL 하나보다 **클라이언트가 예측할 수 있는 규칙 세트**를 먼저 만드는 쪽이 낫습니다.

## 먼저 맞춰둘 체크리스트

- 자원 중심 URL인가
- HTTP 메서드 의미가 맞는가
- 상태 코드가 일관적인가
- 응답 구조가 예측 가능한가
- 오류 응답도 통일돼 있는가
- 페이징/정렬 규칙이 일정한가
- 같은 요청을 다시 보내도 안전한지 기준이 있는가

## 자주 생기는 문제

### 1. 동사형 URL이 많아지는 경우

`/getUsers`, `/createOrder`처럼 동작 중심으로 흘러가면 자원 구조가 금방 흐려집니다.

예를 들어 아래처럼 자원을 기준으로 잡는 편이 더 오래 갑니다.

- `GET /users`
- `POST /orders`
- `PATCH /orders/{id}`

중요한 건 REST 원칙 암기보다, URL만 봐도 무엇을 다루는지 읽히는가입니다.

### 2. 성공과 실패 응답 구조가 뒤섞이는 경우

어떤 API는 바로 데이터를 주고, 어떤 API는 `data`, `result`, `message`를 감싸 주면 클라이언트 처리 코드가 금방 지저분해집니다.

특히 오류 응답 구조가 제각각이면 프런트엔드나 다른 소비자는 예외 케이스를 계속 따로 처리해야 합니다.

### 3. 상태 코드를 거의 200만 쓰는 경우

검증 실패, 권한 없음, 리소스 없음까지 전부 200으로 반환하면 API는 동작해도 운영과 디버깅이 불편해집니다.

이 경우는 화면은 돌아가더라도, 로그와 모니터링에서 실패 흐름이 잘 안 보이는 문제가 생깁니다.

### 4. 목록 API마다 페이징 규칙이 달라지는 경우

어떤 곳은 `page`, 어떤 곳은 `offset`, 어떤 곳은 정렬 파라미터 이름도 다르면 클라이언트가 API마다 별도 코드를 가져가야 합니다.

## 실무에서 특히 중요한 기준

### URL보다 응답 규칙이 더 오래 간다

URL은 나중에 바꿀 기회가 있어도, 응답 구조와 오류 형식은 여러 클라이언트에 퍼지기 시작하면 바꾸기 훨씬 어렵습니다.

### 같은 종류의 API는 같은 규칙을 써야 한다

예를 들어 목록 조회 API라면 아래 정도는 고정하는 편이 좋습니다.

- `page`, `size` 같은 파라미터 이름
- 정렬 방식
- 빈 결과 처리 방식
- 공통 메타데이터 구조

### 멱등성은 재시도 관점에서 본다

특히 결제, 주문, 외부 연동처럼 재시도가 일어날 수 있는 요청은 "같은 요청이 두 번 오면 어떤 일이 생기나"를 같이 봐야 합니다.

이 기준이 없으면 네트워크 재시도나 중복 호출 때 사고가 나기 쉽습니다.

## 간단한 기준 예시

생성 API라면 보통 아래 정도를 기본으로 삼을 수 있습니다.

- 생성 성공: `201 Created`
- 잘못된 요청: `400 Bad Request`
- 권한 없음: `403 Forbidden`
- 없는 리소스 참조: `404 Not Found`
- 충돌 상황: `409 Conflict`

정답 하나를 외우는 것보다, 팀 안에서 이 기준을 예외 없이 유지하는 편이 훨씬 중요합니다.

## 설계가 애매할 때 다시 보는 질문

- 클라이언트가 이 API를 예측 가능하게 쓸 수 있는가
- 실패했을 때 상태 코드와 오류 본문만 보고 대응 가능한가
- 같은 종류의 API와 규칙이 충돌하지 않는가
- 재시도나 중복 호출이 들어와도 의도가 보존되는가

## 정리

- 좋은 API는 화려함보다 일관성이 강하다.
- URL, 메서드, 상태 코드, 응답 형식은 따로가 아니라 같이 봐야 한다.
- 특히 오류 응답, 페이징 규칙, 멱등성 기준은 초반에 맞춰두는 편이 비용이 적다.
- 설계가 애매하면 항상 "클라이언트가 예측 가능하게 쓸 수 있는가"로 다시 보면 정리가 잘 된다.
