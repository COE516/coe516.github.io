## 문제 상황

API를 만들다 보면 URL 이름은 신경 쓰는데, 정작 상태 코드, 응답 구조, 멱등성 같은 중요한 부분은 뒤늦게 맞추는 경우가 많습니다.

처음에는 작은 차이처럼 보여도 API 수가 늘어나면 클라이언트는 매번 다른 규칙을 기억해야 하고, 운영 중 오류 분석도 더 어려워집니다.

## 먼저 맞춰야 할 체크리스트

아래는 초반에 꼭 맞춰두는 편이 좋은 기준입니다.

- 자원 중심 URL인가
- HTTP 메서드 의미가 맞는가
- 상태 코드가 일관적인가
- 응답 구조가 예측 가능한가
- 오류 응답도 통일돼 있는가
- 페이징/정렬 규칙이 일정한가

## 자주 생기는 문제

### 1. 동사형 URL이 많아지는 경우

`/getUsers`, `/createOrder`처럼 API 이름이 동작 중심으로 흘러가면 자원 구조가 금방 흐려집니다.

### 2. 성공과 실패 응답 구조가 뒤섞이는 경우

어떤 API는 바로 데이터를 주고, 어떤 API는 `data`, `result`, `message`를 감싸 주면 클라이언트 처리 코드가 복잡해집니다.

### 3. 상태 코드를 거의 200만 쓰는 경우

검증 실패, 권한 없음, 리소스 없음까지 전부 200으로 반환하면 API는 동작해도 운영과 디버깅이 불편해집니다.

## 실무에서 특히 중요한 기준

### URL보다 응답 규칙이 더 오래 간다

URL은 바꿀 기회가 있어도 응답 구조와 오류 형식은 여러 클라이언트에 퍼지기 시작하면 바꾸기 더 어렵습니다.

### 같은 종류의 API는 같은 규칙을 써야 한다

예를 들어 목록 조회 API라면

- page, size 파라미터 이름
- 정렬 방식
- 빈 결과 처리

같은 기준을 유지하는 편이 훨씬 낫습니다.

## 간단한 예

예를 들어 생성 API라면 아래처럼 맞추는 편이 자연스럽습니다.

- 생성 성공: `201 Created`
- 잘못된 요청: `400 Bad Request`
- 없는 리소스 참조: `404 Not Found`

핵심은 완벽한 정답보다, 같은 규칙을 계속 유지하는 것입니다.

## 정리

- 좋은 API는 화려함보다 일관성이 강하다.
- URL, 메서드, 상태 코드, 응답 형식은 같이 봐야 한다.
- 특히 오류 응답과 페이징 규칙은 초반에 맞춰두는 편이 비용이 적다.
- 설계가 애매하면 "클라이언트가 이 API를 예측 가능하게 쓸 수 있는가"로 다시 보면 정리가 잘 된다.
