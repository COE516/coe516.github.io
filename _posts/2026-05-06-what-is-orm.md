## 문제 상황

백엔드 개발을 시작하면 자주 듣는 말 중 하나가 `ORM을 쓴다`는 표현입니다.

그런데 막상 처음 접하면 ORM이 라이브러리 이름인지, SQL을 아예 몰라도 되게 해주는 도구인지, 아니면 JPA와 같은 뜻인지 헷갈리기 쉽습니다.

실무에서는 ORM을 "DB를 몰라도 되게 해주는 마법"처럼 이해하면 오히려 더 자주 막힙니다.

## ORM이란 무엇인가

ORM은 **Object Relational Mapping**의 약자입니다.

애플리케이션의 객체와 관계형 데이터베이스의 테이블을 서로 연결해 주는 방식 또는 기술을 말합니다.

예를 들어 자바 코드에서는 `User` 객체를 다루고 있지만, 데이터베이스에는 `users` 테이블이 있습니다.
ORM은 이 둘 사이를 매핑해서 객체를 저장하면 테이블에 반영되고, 테이블 데이터를 읽으면 객체로 다룰 수 있게 도와줍니다.

## ORM이 해주는 일

ORM이 있으면 개발자는 매번 SQL 결과를 객체에 수동으로 옮기는 작업을 줄일 수 있습니다.

보통은 아래 같은 역할을 맡습니다.

- 객체와 테이블 간 매핑
- INSERT, UPDATE, DELETE 같은 기본 저장 작업 처리
- 조회 결과를 객체로 변환
- 연관관계 표현
- 트랜잭션 문맥 안에서 변경 감지 지원

즉 ORM은 **데이터 접근을 객체 중심으로 다루기 쉽게 만드는 도구**에 가깝습니다.

## 예를 들면 이런 식이다

예를 들어 회원 정보를 저장한다고 하면 직접 SQL을 다루는 방식은 이런 느낌입니다.

```java
String sql = "select id, name, email from users where id = ?";
PreparedStatement ps = connection.prepareStatement(sql);
ps.setLong(1, 1L);
ResultSet rs = ps.executeQuery();

if (rs.next()) {
    User user = new User(
        rs.getLong("id"),
        rs.getString("name"),
        rs.getString("email")
    );
}
```

ORM을 쓰면 보통은 객체 기준으로 접근합니다.

```java
@Entity
public class User {
    @Id
    private Long id;

    private String name;
    private String email;
}
```

```java
User user = entityManager.find(User.class, 1L);
```

개발자는 `User` 객체를 중심으로 코드를 작성하고, ORM이 필요한 SQL 생성과 매핑 일부를 대신 처리합니다.

## ORM의 장점

### 생산성이 좋아진다

반복적인 CRUD 코드를 줄이기 쉬워집니다.
특히 엔티티가 많고 기본 저장/조회가 자주 나오는 서비스에서는 효과가 큽니다.

### 객체 중심으로 모델링하기 좋다

비즈니스 로직이 객체 구조를 따라가므로, 도메인 모델과 코드 흐름을 맞추기 편해집니다.

### 변경에 덜 취약한 부분이 있다

컬럼 추가나 객체 필드 변경이 생겼을 때, 매핑 정보 기준으로 정리할 수 있어 수동 매핑 코드가 흩어지는 문제를 줄일 수 있습니다.

## ORM을 쓴다고 SQL을 몰라도 되는 것은 아니다

이 지점이 가장 중요합니다.

ORM은 SQL을 없애는 도구가 아니라, **SQL을 직접 다루는 양을 줄여 주는 도구**에 가깝습니다.

실무에서 ORM을 써도 여전히 알아야 하는 것들이 있습니다.

- 어떤 쿼리가 실제로 나가는지
- 조인이 많을 때 성능이 어떻게 달라지는지
- N+1 문제가 왜 생기는지
- 인덱스와 실행 계획이 왜 중요한지
- 트랜잭션 경계 안에서 어떤 시점에 flush가 일어나는지

즉 ORM을 잘 쓰려면 객체만 보는 것이 아니라, 그 뒤에서 실행되는 SQL도 같이 볼 줄 알아야 합니다.

## 언제 잘 맞을까

ORM은 보통 아래 같은 상황에서 잘 맞습니다.

- 전형적인 CRUD 비중이 큰 서비스
- 도메인 모델 중심으로 개발하고 싶은 경우
- 반복적인 매핑 코드를 줄이고 싶은 경우
- 저장/수정 로직과 객체 상태 관리가 중요한 경우

반대로 아래 같은 경우는 ORM만으로 밀어붙이기보다 다른 접근을 같이 고려할 수 있습니다.

- 매우 복잡한 통계성 조회가 많은 경우
- SQL 튜닝과 쿼리 제어가 핵심인 경우
- 화면 전용 조회가 매우 다양한 경우

그래서 실무에서는 ORM 하나로 모든 것을 해결하기보다, 기본 도메인 작업은 ORM으로 처리하고 복잡한 조회는 별도 SQL 방식으로 분리하기도 합니다.

## JPA와 Hibernate는 ORM과 어떤 관계인가

이 부분도 자주 헷갈립니다.

- ORM: 객체와 테이블을 매핑하는 개념 또는 접근 방식
- JPA: 자바 진영의 ORM 표준 명세
- Hibernate: JPA를 구현하는 대표적인 구현체

즉 ORM은 더 큰 개념이고, JPA와 Hibernate는 그것을 자바에서 사용하기 위한 표준과 구현이라고 보면 이해가 쉽습니다.

## 정리

- ORM은 객체와 관계형 데이터베이스를 연결해 주는 기술이다.
- 반복적인 데이터 접근 코드를 줄이고 객체 중심 개발을 돕는다.
- 하지만 ORM을 쓴다고 데이터베이스와 SQL을 몰라도 되는 것은 아니다.
- 실무에서는 ORM의 편의성과 SQL 이해를 함께 가져가는 것이 중요하다.
