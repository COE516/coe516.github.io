## 문제 상황

정산, 집계, 대량 데이터 처리 같은 작업을 만들 때 자주 드는 질문이 있습니다.

"스케줄러 하나로 직접 돌려도 되나, 아니면 Spring Batch까지 가야 하나?"

이 판단은 작업이 크냐 작냐보다 **실패했을 때 어떻게 복구해야 하느냐**로 보는 편이 더 실무적입니다.

## 직접 구현이 더 나은 경우

아래 조건이면 보통은 직접 구현으로도 충분합니다.

- 작업 흐름이 단순하다.
- 실패 시 처음부터 다시 돌려도 부담이 크지 않다.
- 처리 대상이 아주 크지 않다.
- 실행 이력이나 step 단위 추적이 필수는 아니다.

예를 들어 매일 통계 테이블을 다시 집계하는 정도라면 `@Scheduled`와 서비스 로직만으로도 충분히 운영 가능한 경우가 많습니다.

## Spring Batch가 값을 하는 경우

반대로 아래 요구사항이 붙기 시작하면 Spring Batch 쪽이 점점 유리해집니다.

- 실패 지점부터 재시작해야 한다.
- 대량 데이터를 chunk 단위로 끊어 처리해야 한다.
- skip, retry 같은 제어가 필요하다.
- 실행 이력과 상태를 남겨야 한다.
- step 단위로 작업을 나누고 운영에서 추적해야 한다.

핵심은 코드 양이 아니라 **운영 기능을 프레임워크가 대신 가져가 주는가**입니다.

## 가장 먼저 볼 질문

판단이 애매하면 아래 질문 네 개로 많이 정리됩니다.

1. 실패하면 처음부터 다시 돌려도 되는가
2. 처리 중간 상태를 남겨야 하는가
3. 재시도, 건너뛰기, chunk 처리가 필요한가
4. 운영자가 실행 이력을 직접 봐야 하는가

이 질문에 "예"가 많을수록 Spring Batch 쪽으로 기울 가능성이 높습니다.

## 흔한 오해

### 작업이 크면 무조건 Batch를 써야 한다?

그렇지는 않습니다. 작업량이 커도 흐름이 단순하고 재실행 비용이 낮다면 직접 구현이 더 단순할 수 있습니다.

### 단순 스케줄 작업인데도 미리 Batch를 넣어야 한다?

이것도 아닙니다. 지금 필요한 운영 요구사항이 없는데 Batch를 먼저 넣으면 설정과 개념 비용만 커질 수 있습니다.

결국 기준은 "복잡해 보이니까"가 아니라 **실패 복구와 운영 가시성이 필요한가**입니다.

## 정리

- 단순 주기 작업이면 직접 구현이 더 빠르고 관리하기 쉽다.
- 재시작, chunk, retry, 실행 이력이 중요해지면 Spring Batch가 유리하다.
- 작업 크기보다 실패 복구 방식과 운영 요구사항으로 판단하는 편이 맞다.
- Batch는 미래를 위한 보험이 아니라, 지금 필요한 운영 기능이 있을 때 도입하는 편이 좋다.
