## 문제 상황

작은 프로젝트에서는 Service와 Repository 경계가 쉽게 흐려집니다.

조회 조건 조립, 상태 검증, 트랜잭션 처리, 저장 흐름까지 한쪽으로 몰리면 처음에는 빨라도 나중에 수정 포인트를 찾기 어려워집니다.

특히 기능이 조금만 커지면 "이 로직이 쿼리 책임인지, 업무 책임인지"가 모호해지면서 계층이 빠르게 섞이기 시작합니다.

## 기본 기준

### Repository

Repository는 **DB와 직접 대화하는 책임**을 가집니다.

- 저장
- 조회
- 삭제
- 쿼리 실행

즉 "어떻게 읽고 쓰는가"에 더 가깝습니다.

### Service

Service는 **비즈니스 흐름을 조합하는 책임**을 가집니다.

- 여러 Repository 호출 조합
- 상태 검증
- 트랜잭션 경계
- 업무 규칙 적용

즉 "무엇을 왜 처리하는가"에 더 가깝습니다.

## 판단 질문으로 보면

경계가 애매하면 아래 질문이 꽤 잘 통합니다.

- 이 로직은 DB 접근 기술 세부사항인가
- 여러 저장/조회 흐름을 묶는가
- 실패/성공 조건이 업무 규칙에 묶여 있는가
- 트랜잭션 경계를 설명해야 하는가

앞쪽이면 Repository, 뒤쪽이면 Service일 가능성이 큽니다.

## 예를 들면 이렇게 나눈다

주문 생성 로직을 생각해 보면,

- 주문을 저장한다.
- 재고를 차감한다.
- 결제 상태를 기록한다.

각 저장 쿼리 자체는 Repository 책임입니다.

반면 이 세 단계를 어떤 순서로 묶고, 어떤 조건에서 실패시킬지는 Service 책임에 가깝습니다.

즉 `save()` 자체는 Repository 문제이고, `주문 생성 전체를 어떤 규칙으로 처리할지`는 Service 문제입니다.

## 흔한 실수

### Repository에 비즈니스 분기까지 넣는 경우

조회 조건을 넘어서 승인/거절 판단 같은 업무 규칙이 Repository 안으로 들어가면 계층 경계가 금방 무너집니다.

Repository가 "어떤 데이터를 가져올까"를 넘어서 "이 요청을 승인할까"까지 판단하기 시작하면 나중에 로직 재사용이 어려워집니다.

### Service가 SQL 수준 세부 구현까지 아는 경우

Service에서 쿼리 최적화 방식, 조인 방식, 저장 기술 세부사항까지 신경 쓰기 시작하면 반대로 Repository 책임이 약해집니다.

Service가 SQL 구조를 너무 많이 알면 저장 방식이 바뀔 때 업무 로직까지 같이 흔들리기 쉽습니다.

## 작은 프로젝트에서 더 무너지기 쉬운 이유

초기에는 파일 수를 줄이는 쪽이 빨라 보일 수 있습니다.

하지만 기능이 늘어나면 수정 포인트가 Service인지 Repository인지 불분명해지고, 한쪽 수정이 다른 쪽 규칙까지 건드리기 시작합니다. 그래서 작은 프로젝트일수록 초기에 책임을 가볍게라도 나눠 두는 편이 오히려 편할 때가 많습니다.

## 정리

- Repository는 데이터 접근 책임을 가진다.
- Service는 비즈니스 흐름과 규칙 책임을 가진다.
- 저장 기술 문제면 Repository, 업무 규칙 문제면 Service 쪽으로 두는 편이 자연스럽다.
- 경계가 애매할수록 "이 로직이 왜 존재하는가"를 기준으로 다시 보면 정리가 빨라진다.
