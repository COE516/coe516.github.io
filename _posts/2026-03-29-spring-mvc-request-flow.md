## 문제 상황

필터는 어디서 실행되는지, 인터셉터는 언제 도는지, 예외 처리는 누가 잡는지 헷갈리는 경우가 많습니다.

특히 인증, 로깅, 공통 예외 처리 위치를 잡을 때 이 흐름을 모르면 매번 감으로 붙이게 됩니다.

## 큰 흐름

Spring MVC 요청은 보통 아래 순서로 봅니다.

1. 클라이언트 요청 도착
2. 서블릿 필터 통과
3. `DispatcherServlet` 진입
4. `HandlerMapping`으로 대상 컨트롤러 탐색
5. `HandlerAdapter`가 컨트롤러 메서드 실행
6. 반환값 처리
7. 뷰 렌더링 또는 JSON 직렬화 후 응답

이 순서를 알고 있으면 "지금 내가 넣으려는 로직이 웹 앞단인지, MVC 내부인지"를 훨씬 빨리 구분할 수 있습니다.

## DispatcherServlet이 중요한 이유

`DispatcherServlet`은 Spring MVC의 중심입니다.

컨트롤러를 직접 부르는 대신,

- 어떤 핸들러를 실행할지 찾고
- 실행 방식을 고르고
- 예외 처리와 응답 변환까지 연결합니다.

즉 요청을 "컨트롤러로 보내는 문"이 아니라, MVC 전체 흐름을 조립하는 진입점에 가깝습니다.

## 필터와 인터셉터는 어디가 다를까

### 필터

서블릿 앞단에서 동작합니다.

- 인코딩 처리
- 공통 로깅
- 인증 토큰 1차 확인

같은 웹 전역 관심사에 잘 맞습니다.

### 인터셉터

Spring MVC 내부에서 동작합니다.

- 로그인 사용자 검사
- 특정 컨트롤러 전후 공통 처리
- 요청 시간 측정

처럼 컨트롤러 실행과 더 가까운 작업에 잘 맞습니다.

## 예외 처리는 어디서 보나

컨트롤러 실행 중 예외가 나면 `@ExceptionHandler`, `@ControllerAdvice` 같은 예외 처리기가 개입합니다.

그래서 예외 응답 형식을 통일하려면 Controller 바깥 필터보다 MVC 내부 예외 처리기를 먼저 떠올리는 편이 맞는 경우가 많습니다.

## 디버깅할 때 보는 순서

요청이 이상하게 동작하면 보통 이 순서로 좁혀갑니다.

- 필터에서 이미 막혔는가
- 인터셉터에서 차단됐는가
- 컨트롤러까지 진입했는가
- 예외 처리기로 넘어갔는가
- 응답 변환 단계에서 깨졌는가

이렇게 보면 필터/인터셉터/예외 처리 위치가 훨씬 덜 섞입니다.

## 정리

- 필터는 서블릿 앞단, 인터셉터는 Spring MVC 내부다.
- `DispatcherServlet`은 요청 처리 전체를 조립하는 중심이다.
- 인증, 로깅, 예외 처리 위치를 정할 때 이 흐름을 기준으로 보면 덜 헷갈린다.
