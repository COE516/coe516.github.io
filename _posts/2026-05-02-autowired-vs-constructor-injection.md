## 문제 상황

예전 코드에는 필드 주입이 많고, 새 코드에서는 생성자 주입을 권장하는 경우가 많습니다. 그냥 스타일 차이처럼 보이지만 실무에서는 클래스 구조와 테스트 편의성에 꽤 직접적인 차이를 만듭니다.

핵심은 주입 문법보다 **의존성이 코드에 얼마나 분명하게 드러나느냐**입니다.

## 생성자 주입이 기본이 되는 이유

가장 큰 장점은 필요한 의존성이 명확하게 보인다는 점입니다.

```java
@Service
public class OrderService {
    private final PaymentClient paymentClient;

    public OrderService(PaymentClient paymentClient) {
        this.paymentClient = paymentClient;
    }
}
```

이 클래스가 무엇 없이는 동작할 수 없는지 바로 드러나고, `final`로 고정하기도 쉽습니다.

## 필드 주입이 아쉬운 이유

- 생성 시 필요한 의존성이 코드에 덜 드러남
- `final` 사용이 어려움
- 테스트에서 대체 객체를 넣기 불편함
- 클래스가 커져도 의존성이 늘어나는 것을 놓치기 쉬움

즉 처음엔 짧아 보여도 구조를 읽는 입장에서는 불리한 경우가 많습니다.

## 실무에서 자주 갈리는 부분

### 모든 걸 생성자로만 넣어야 하나

대부분의 필수 의존성은 생성자 주입이 가장 무난합니다. 다만 선택 의존성이나 지연 조회가 필요한 경우는 다른 방식이 더 자연스러울 수 있습니다.

### `@Autowired`를 아예 쓰면 안 되나

요즘은 단일 생성자라면 `@Autowired`를 생략하는 경우가 많습니다. 중요한 건 애노테이션 유무보다 생성자 기반 구조를 쓰는지입니다.

### 테스트가 왜 더 편해지나

생성자 주입은 필요한 의존성을 명시적으로 넘겨 객체를 만들 수 있어서 단위 테스트에서 훨씬 다루기 쉽습니다.

## 짧은 기준

- 새 코드는 생성자 주입을 기본으로 본다.
- 필수 의존성은 생성자에서 받는다.
- 선택 의존성은 정말 필요할 때만 별도 방식으로 본다.
- 핵심은 코드 길이보다 의존성을 숨기지 않는 것이다.

## 정리

- 생성자 주입은 의존성을 명확하게 드러낸다.
- 필드 주입은 짧지만 구조와 테스트 측면에서 아쉬움이 크다.
- 요즘 기준에서는 생성자 주입이 가장 무난한 기본 선택이다.
- 주입 방식은 취향보다 클래스 책임을 얼마나 분명하게 보여주느냐의 문제에 가깝다.
