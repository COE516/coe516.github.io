## 왜 자주 헷갈릴까

면접이나 CS 공부에서는 보통 이렇게 외웁니다.

- 프로세스는 실행 중인 프로그램
- 스레드는 프로세스 안에서 실행되는 작업 흐름

정의만 보면 맞는 말이지만, 실무에서는 이 정도로는 부족합니다.

왜냐하면 실제 문제는 "메모리를 누구와 공유하느냐", "하나가 죽었을 때 어디까지 같이 영향을 받느냐", "동시에 여러 일을 시킬 때 비용이 얼마나 드느냐"에서 갈리기 때문입니다.

## 한 줄로 보면

- **프로세스**: 독립된 메모리 공간을 가진 실행 단위
- **스레드**: 같은 프로세스 안에서 메모리를 공유하며 실행되는 단위

핵심 차이는 결국 **격리**와 **공유**입니다.

## 가장 큰 차이: 메모리 공유 여부

프로세스는 서로 메모리를 기본적으로 공유하지 않습니다.

예를 들어 브라우저와 IDE는 각각 별도 프로세스로 떠 있고, 한쪽이 비정상 종료되어도 다른 쪽 메모리가 직접 같이 깨지지는 않습니다.

반면 스레드는 같은 프로세스 안의 힙 메모리, 전역 변수 같은 자원을 함께 사용합니다.

그래서 같은 데이터를 빠르게 주고받기 쉽지만, 동시에 접근하면 동기화 문제가 생길 수 있습니다.

```java
public class Counter {
    private int count = 0;

    public void increment() {
        count++;
    }
}
```

여러 스레드가 `increment()`를 동시에 호출하면 `count++`가 안전하지 않을 수 있습니다.

즉 스레드는 가볍고 빠르지만, 공유 자원 관리 책임이 따라옵니다.

## 장애 영향 범위도 다르다

프로세스는 서로 격리되어 있어서 한 프로세스가 죽어도 다른 프로세스까지 바로 같이 죽지는 않는 경우가 많습니다.

스레드는 다릅니다. 같은 프로세스 안에서 동작하므로 잘못된 공유 메모리 접근이나 치명적인 오류가 나면 프로세스 전체 안정성에 영향을 줄 수 있습니다.

실무에서 이 차이는 꽤 중요합니다.

- 작업 간 강한 격리가 필요하면 프로세스 분리가 유리할 수 있고
- 빠른 협업과 낮은 통신 비용이 중요하면 스레드 기반 처리가 유리할 수 있습니다.

## 생성 비용과 문맥 전환 비용

보통 프로세스는 스레드보다 무겁습니다.

새 프로세스를 띄우면 독립된 메모리 공간과 운영체제 자원이 더 필요하고, 전환 비용도 더 큽니다.

반대로 스레드는 같은 프로세스 내부 자원을 활용하므로 생성과 전환이 비교적 가볍습니다.

그래서 웹 서버, 비동기 처리, 병렬 작업에서는 멀티 스레드가 자주 쓰입니다.

다만 "가볍다"가 "무한정 늘려도 된다"는 뜻은 아닙니다.

스레드 수가 과도해지면 문맥 전환 비용이 커지고, 락 경쟁까지 겹치면서 오히려 더 느려질 수 있습니다.

## 통신 방식도 다르다

프로세스끼리는 기본적으로 메모리를 직접 공유하지 않으므로 통신하려면 별도 수단이 필요합니다.

예를 들면 이런 방식입니다.

- 소켓
- 파이프
- 메시지 큐
- 공유 메모리

스레드는 같은 메모리를 공유하므로 데이터 전달 자체는 더 단순합니다.

대신 단순한 만큼 경쟁 상태, 가시성, 락 설계 같은 문제가 따라옵니다.

즉 **프로세스는 통신이 불편하고**, **스레드는 동기화가 어렵다**고 보면 이해가 쉽습니다.

## 실무에서 어떻게 구분해서 생각하면 좋을까

### 1. 서로 완전히 분리돼야 하는가

장애 격리, 보안 경계, 리소스 독립성이 중요하면 프로세스 분리가 더 잘 맞습니다.

예를 들어 서로 다른 서비스, 브라우저 탭 격리, 워커 프로세스 분리 같은 경우입니다.

### 2. 같은 데이터를 자주 같이 써야 하는가

짧은 작업을 빠르게 주고받아야 하고 메모리 공유 이점이 크다면 스레드가 더 적합할 수 있습니다.

대표적으로 웹 요청 처리, 백그라운드 작업, 병렬 계산이 그렇습니다.

### 3. 문제의 성격이 CPU 바운드인가 I/O 바운드인가

- I/O 대기가 많으면 스레드 기반 동시성으로 효율이 좋아질 수 있고
- CPU를 강하게 쓰는 작업은 스레드 수만 늘린다고 해결되지 않을 수 있습니다.

이 경우는 런타임 특성, 코어 수, 프로세스 분리 전략까지 같이 봐야 합니다.

## 자주 나오는 오해

### "멀티 스레드면 무조건 빠르다"

아닙니다. 공유 자원 충돌, 락 대기, 문맥 전환 비용이 커지면 단일 스레드보다 더 느릴 수도 있습니다.

### "프로세스는 무조건 안전하다"

메모리 격리 측면에서는 유리하지만, IPC 복잡도와 자원 사용량은 더 커질 수 있습니다.

### "스레드는 함수 호출 비슷한 수준이다"

생성 비용이 프로세스보다 낮을 뿐이고, 스케줄링과 동기화 문제는 여전히 운영체제 수준에서 중요합니다.

## 정리

프로세스와 스레드 차이는 결국 아래 네 가지로 정리할 수 있습니다.

- 프로세스는 메모리가 분리되고, 스레드는 메모리를 공유한다.
- 프로세스는 격리에 유리하고, 스레드는 협업과 속도에 유리하다.
- 프로세스 통신은 추가 수단이 필요하고, 스레드는 동기화가 중요하다.
- 스레드는 가볍지만 많이 쓰면 오히려 병목이 될 수 있다.

정의를 외우는 것보다 **공유 자원**, **장애 전파 범위**, **전환 비용** 관점으로 보면 실무에서도 덜 헷갈립니다.
