## 문제 상황

운영에서는 민감한 값을 `application.yml`에 직접 넣기보다 환경변수로 주입하는 경우가 많습니다.

하지만 실제로는 단순히 값을 넣는 것보다 아래에서 자주 헷갈립니다.

- 이름을 어떻게 맞추는지
- profile 설정과 함께 쓸 때 무엇이 최종값이 되는지
- 로그나 설정 출력에서 민감한 값이 새지 않는지

## 기본 방식

Spring Boot는 환경변수를 프로퍼티와 자동으로 매핑합니다.

예를 들어 아래 프로퍼티는:

```yaml
app:
  api-key: test
```

환경변수에서는 보통 이렇게 대응됩니다.

```bash
APP_API_KEY=real-key
```

즉 점(`.`)과 하이픈(`-`)이 언더스코어 기반 이름으로 바뀌는 식으로 이해하면 됩니다.

## 왜 운영에서 많이 쓰일까

- 배포 환경별 교체가 쉬움
- 비밀값을 코드 저장소 밖으로 뺄 수 있음
- 컨테이너 환경과 잘 맞음
- 동일 이미지를 여러 환경에 재사용하기 편함

특히 Docker나 Kubernetes처럼 실행 환경에서 값을 주입하는 구조와 잘 맞습니다.

## 같이 봐야 할 포인트

### 1. 이름 규칙을 정확히 맞춰야 한다

오타가 나면 애플리케이션은 떠도 실제 값이 안 들어간 채 기본값으로 동작할 수 있습니다.

### 2. 우선순위를 확인해야 한다

`application.yml`, profile별 설정, 환경변수가 함께 있을 때는 어떤 값이 최종적으로 적용되는지 확인해야 합니다.

실무에서는 "분명 yml을 바꿨는데 왜 값이 안 바뀌지" 같은 상황이 여기서 자주 나옵니다.

### 3. 민감한 값 노출을 조심해야 한다

로그, actuator, 디버그 출력에서 비밀값이 그대로 보이지 않는지 같이 봐야 합니다.

## 실무 기준

- 공통 기본값은 yml에 둔다.
- 환경별 민감값은 환경변수로 분리한다.
- 설정이 안 먹는 것 같으면 우선순위와 실제 주입 이름부터 본다.

이 정도 기준만 잡아도 운영 설정 실수가 꽤 줄어듭니다.

## 정리

- 환경변수 주입은 운영에서 가장 자주 쓰는 방식 중 하나다.
- 핵심은 값 자체보다 `이름 매핑`, `우선순위`, `민감값 보호`다.
- 설정 문제가 생기면 yml보다 먼저 실제 주입된 환경변수와 최종 적용값을 확인하는 편이 빠르다.
