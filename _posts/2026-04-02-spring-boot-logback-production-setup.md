## 문제 상황

처음에는 콘솔 로그만으로도 충분해 보이지만, 운영에 들어가면 요청 추적, 에러 확인, 검색 편의성 때문에 로그 구성이 중요해집니다.

로그를 많이 남기는 것 자체보다, **필요한 로그를 운영 중에 다시 찾을 수 있게 남기는 것**이 더 중요합니다.

## 실무에서 자주 하는 실수

- 모든 로그를 DEBUG로 남김
- 콘솔과 파일 로그를 구분하지 않음
- 패턴에 요청 식별자 없음
- 운영 로그 롤링 정책 없음

이렇게 두면 로그가 많아질수록 읽기 어려워지고, 장애 대응 시에도 필요한 흐름을 다시 찾기 힘들어집니다.

## 기본 기준

실무에서는 보통 아래를 먼저 나눕니다.

- 로컬: 콘솔 위주
- 운영: 파일 또는 수집 시스템 위주
- 로그 레벨: 기본 INFO, 필요한 패키지만 DEBUG

즉 환경마다 "무엇을 보기 위해 로그를 남기는가"부터 달라집니다.

## 간단한 예

```xml
<springProfile name="prod">
    <root level="INFO">
        <appender-ref ref="FILE" />
    </root>
</springProfile>
```

프로파일별로 출력을 나누면 개발용 로그와 운영 로그가 섞이지 않습니다.

## 꼭 챙길 것

### 1. 롤링 정책

하루 단위 또는 용량 단위로 끊어야 합니다. 이게 없으면 파일이 커지기만 하고 검색과 보관이 모두 불편해집니다.

### 2. 요청 식별자

MDC를 써서 trace id나 request id를 남기면 장애 추적이 쉬워집니다.

### 3. 패키지별 레벨 조정

전체 DEBUG보다 필요한 패키지만 올리는 편이 훨씬 낫습니다. 특히 운영에서는 루트 로그 레벨을 낮게 유지하는 편이 안전합니다.

## 실무 기준

- 로컬은 읽기 쉬운 콘솔 로그
- 운영은 수집/검색 가능한 형식
- 기본 INFO, 필요한 패키지만 DEBUG
- 요청 식별자와 롤링 정책은 초반부터 같이 본다

## 정리

- 로그 설정은 많이 남기는 것보다 읽히게 남기는 것이 중요하다.
- 로컬과 운영 로그는 목적이 다르니 분리하는 편이 낫다.
- 요청 식별자, 롤링 정책, 레벨 조정이 실제 운영 품질을 많이 좌우한다.
- 운영 장애 대응을 생각하면 출력 방식보다 먼저 검색 가능성과 추적 가능성을 기준으로 잡는 편이 좋다.
