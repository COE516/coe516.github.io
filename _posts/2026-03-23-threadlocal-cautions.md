## 문제 상황

요청별 사용자 정보나 trace id를 잠깐 담아두려고 `ThreadLocal`을 쓰는 경우가 많습니다.

문제는 로컬 테스트에서는 멀쩡해 보여도, 운영에서는 스레드풀이 재사용되면서 이전 요청의 값이 다음 요청에 섞일 수 있다는 점입니다. `set()`은 했는데 `remove()`를 빼먹은 경우가 대표적입니다.

## 왜 스레드풀에서 더 위험할까

`ThreadLocal`의 값은 **요청에 붙는 것이 아니라 스레드에 붙습니다.**

그래서 톰캣 같은 서버가 같은 워커 스레드를 재사용하면, 이전 요청에서 남긴 값이 다음 요청에서도 그대로 보일 수 있습니다.

이럴 때 흔히 보이는 증상은 아래와 같습니다.

- 다른 사용자의 trace id가 로그에 섞임
- 테스트에서는 재현이 어려운데 운영에서만 간헐적으로 값이 꼬임
- 메모리 사용량이 조금씩 늘어나는데 원인을 찾기 어려움

## 가장 중요한 규칙은 `remove()`다

`ThreadLocal`을 직접 써야 한다면 저장보다 정리를 먼저 생각하는 편이 안전합니다.

```java
private static final ThreadLocal<String> TRACE_ID = new ThreadLocal<>();

public void handle(String traceId) {
    try {
        TRACE_ID.set(traceId);
        process();
    } finally {
        TRACE_ID.remove();
    }
}
```

실무에서는 `set()`을 여러 군데에 흩뿌리기보다, 요청 시작과 종료가 분명한 지점 한 곳에서 넣고 지우는 쪽이 훨씬 낫습니다.

## 비동기 작업으로 넘어가면 값이 이어진다고 기대하면 안 된다

`@Async`, 별도 executor, 메시지 소비처럼 스레드가 바뀌는 지점에서는 같은 `ThreadLocal` 값을 그대로 쓸 수 있다고 보면 안 됩니다.

즉, 아래 같은 기대는 쉽게 깨집니다.

- 웹 요청에서 넣은 사용자 정보가 비동기 메서드에서도 그대로 보일 것
- MDC나 trace 정보가 별도 스레드에서도 자동으로 유지될 것

스레드가 바뀌면 컨텍스트 전달도 명시적으로 설계해야 합니다.

## 언제 직접 쓰지 않는 편이 나을까

아래 상황이면 `ThreadLocal`을 직접 다루기보다 프레임워크가 제공하는 컨텍스트를 우선 보는 편이 좋습니다.

- 요청 범위 데이터라면 Spring Web 요청 컨텍스트
- 로깅 문맥이라면 MDC
- 보안 정보라면 Spring Security 컨텍스트

직접 구현할수록 값 생성보다 정리 누락, 전파 누락, 테스트 재현 어려움이 더 큰 비용으로 돌아오는 경우가 많습니다.

## 정리

- `ThreadLocal` 값은 요청이 아니라 스레드에 묶인다.
- 스레드풀 환경에서는 `remove()` 누락이 가장 흔한 사고 지점이다.
- 비동기 경계를 넘으면 값이 자동으로 이어진다고 기대하면 안 된다.
- 직접 쓰더라도 `try/finally`로 넣고 지우는 위치를 한 곳에 모으는 편이 안전하다.
