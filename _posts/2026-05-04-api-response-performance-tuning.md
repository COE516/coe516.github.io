## 문제 상황

API 응답이 느릴 때 바로 캐시나 코드 최적화부터 떠올리기 쉽습니다.

하지만 실무에서는 병목 위치를 잘못 잡아서 시간을 쓰는 경우가 더 많습니다. ORM 튜닝을 했는데 원인은 외부 API였고, 캐시를 붙였는데 실제로는 응답 본문이 너무 큰 경우도 흔합니다.

그래서 응답 속도 개선은 "무엇을 넣을까"보다 **어디서 느린지 어떤 순서로 좁힐까**가 먼저입니다.

## 보통 이 순서로 본다

### 1. DB 쿼리

가장 먼저 느린 쿼리, N+1, 인덱스 문제를 봅니다.

- 쿼리 시간이 긴가
- 같은 요청에서 쿼리가 과하게 많이 나가는가
- 페이징 없이 큰 범위를 한 번에 읽고 있지는 않은가

애플리케이션 코드보다 DB가 병목인 경우가 여전히 많습니다.

### 2. 외부 API 호출

다음으로 외부 HTTP, 메시지 브로커, 내부 다른 서비스 호출 시간을 봅니다.

- 응답 지연이 특정 외부 시스템 구간에 몰리는가
- 재시도 때문에 실제 대기 시간이 더 길어지지는 않는가
- 요청 하나에서 외부 호출을 여러 번 반복하고 있지는 않은가

애플리케이션 서버가 느린 것이 아니라, 기다리는 시간이 긴 경우가 여기서 많이 드러납니다.

### 3. 직렬화와 응답 본문 크기

쿼리와 외부 호출이 멀쩡한데도 느리다면 응답 payload 크기와 직렬화 비용을 봅니다.

- 너무 많은 필드를 한 번에 내려주지 않는가
- 리스트 응답이 과도하게 큰가
- DTO 구성 과정에서 불필요한 변환을 반복하지 않는가

특히 모바일이나 외부 공개 API는 payload를 줄이는 것만으로 체감이 크게 달라질 수 있습니다.

### 4. 캐시 가능성

캐시는 마지막 수단이라기보다 **조건이 맞을 때 가장 빠른 해법**입니다.

다만 먼저 확인할 것은 아래입니다.

- 데이터가 자주 바뀌는가
- 약간의 stale 허용이 가능한가
- 캐시 무효화 기준을 설명할 수 있는가

병목 위치를 모른 채 캐시부터 붙이면 문제를 가리기만 하고 운영 복잡도만 늘 수 있습니다.

## 측정 없이 튜닝하지 않는 이유

응답 속도 문제는 여러 구간이 조금씩 느린 경우보다, 한 구간이 압도적으로 느린 경우가 더 많습니다.

그래서 아래 정도만 잡아도 방향이 빨리 보입니다.

- 요청 전체 시간
- DB 쿼리 총 시간
- 외부 호출 시간
- 응답 크기

이 네 개를 모른 채 이것저것 최적화하면, 고생은 많이 하고 효과는 거의 없는 경우가 많습니다.

## 정리

- 응답 속도 개선은 기법보다 병목 탐색 순서가 중요하다.
- 보통은 DB, 외부 호출, 직렬화와 payload, 캐시 순으로 좁혀보면 빠르다.
- 캐시는 강력하지만 병목 위치를 확인한 뒤 붙이는 편이 안전하다.
- 먼저 한 구간을 정확히 느리다고 말할 수 있어야 튜닝도 의미가 생긴다.
