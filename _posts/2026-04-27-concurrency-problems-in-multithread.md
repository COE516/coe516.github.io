## 문제 상황

로컬에서는 잘 되는데 동시에 요청이 몰리면 재고가 음수가 되거나, 중복 처리, 상태 꼬임이 발생할 수 있습니다.

이럴 때 무조건 `synchronized`나 락부터 떠올리기 쉽지만, 실제로는 **어디에서 같은 데이터를 동시에 건드리는지**부터 나누는 편이 먼저입니다.

## 왜 생길까

여러 스레드나 여러 인스턴스가 같은 자원을 동시에 읽고 바꾸기 때문입니다.

예를 들어 재고 1개를 남긴 상태에서 두 요청이 동시에 들어오면,

- 둘 다 재고가 있다고 읽고
- 둘 다 차감 로직을 실행해
- 결과가 꼬일 수 있습니다.

즉 핵심은 "동시에 같은 상태를 읽고, 그 상태를 기준으로 수정한다"는 점입니다.

## 먼저 나눠서 봐야 할 것

실무에서는 보통 충돌 지점을 먼저 구분합니다.

- 애플리케이션 메모리 수준인가
- 단일 DB 행 수준인가
- 여러 인스턴스 간 문제인가

이 구분이 안 되면 락 종류를 바꿔도 해결이 덜 됩니다.

## 보통 쓰는 해결책

### 1. 낙관적 락

충돌이 자주 나지 않는 경우에 잘 맞습니다.

버전 충돌이 발생하면 재시도하거나 사용자에게 다시 시도하게 만드는 방식입니다.

### 2. 비관적 락

같은 데이터를 동시에 수정할 가능성이 크고, 충돌 비용이 큰 경우에 더 잘 맞습니다.

대신 대기 시간이 길어질 수 있어 트래픽이 큰 구간에서는 주의가 필요합니다.

### 3. 분산 락

애플리케이션 인스턴스가 여러 대일 때 단일 서버 기준 락으로는 부족할 수 있습니다.

이 경우 Redis 같은 외부 저장소 기반 락을 검토하게 됩니다.

### 4. 큐 기반 직렬화

동시에 처리하지 말고 아예 순서대로 처리하게 만드는 방식이 더 단순한 경우도 있습니다.

주문, 정산, 포인트 차감처럼 순차성이 중요한 흐름에서 자주 검토합니다.

## 어떻게 고를까

### 애플리케이션 메모리 수준 문제

단일 인스턴스 안에서만 생기는 문제라면 코드 레벨 동기화가 통할 수 있습니다.

### DB 행 충돌 문제

같은 row를 동시에 수정하는 문제라면 낙관적 락/비관적 락을 먼저 봅니다.

### 여러 인스턴스 간 문제

서버가 여러 대라면 단일 JVM 락으로는 부족할 수 있어 분산 락이나 큐를 같이 봐야 합니다.

## 정리

- 동시성 문제는 재현보다 충돌 지점 분리가 먼저다.
- 같은 서버 내부 문제인지, DB 충돌인지, 분산 환경 문제인지 먼저 나눠야 한다.
- 충돌 지점에 맞는 락 전략을 골라야 한다.
- 단일 서버 기준 해결책을 분산 환경에 그대로 가져가면 부족할 수 있다.
